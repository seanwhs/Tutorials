# Appendix I.3.7 — Third-Party, Supply Chain & Digital Ecosystem Resilience Assessment Workbook

> *"Your organization's resilience extends only as far as the resilience of the ecosystem on which it depends."*

---

# Purpose

Modern enterprises rarely operate in isolation.

Critical business services depend upon:

* Cloud service providers.
* Software-as-a-Service (SaaS) platforms.
* Managed Service Providers (MSPs).
* Managed Security Service Providers (MSSPs).
* Business partners.
* Software vendors.
* Open-source software.
* APIs.
* Data providers.
* Fourth-party and downstream suppliers.

A disruption affecting any of these entities can quickly become an enterprise crisis.

This workbook evaluates whether the organization understands, governs, monitors, and improves resilience across its extended digital ecosystem.

---

# Assessment Scope

This workbook evaluates:

* Third-party governance.
* Supplier due diligence.
* Critical supplier identification.
* Cloud and SaaS resilience.
* Software supply chain security.
* Fourth-party dependencies.
* Concentration risk.
* Continuous supplier assurance.
* Supplier incident response.
* Ecosystem resilience improvement.

---

# Capability Objectives

A mature ecosystem resilience capability ensures that:

✓ Critical suppliers are identified and prioritized.

✓ Supplier risks are assessed using business impact.

✓ Software supply chain risks are actively managed.

✓ Cloud and SaaS dependencies are understood.

✓ Contractual resilience expectations are defined.

✓ Supplier disruptions can be managed without unacceptable business impact.

# TP1 — Third-Party Governance

## Objective

Assess governance of external dependencies.

### Assessment Areas

Evaluate:

* Third-party risk policy.
* Governance structure.
* Business ownership.
* Procurement integration.
* Executive oversight.
* Risk reporting.

### Sample Questions

| Question                                                     | Score |
| ------------------------------------------------------------ | :---: |
| Third-party governance policies are documented and approved. |       |
| Business owners are accountable for supplier risk.           |       |
| Procurement includes cyber resilience requirements.          |       |
| Governance committees review supplier risks regularly.       |       |
| Critical suppliers are reviewed by executive leadership.     |       |

### Evidence

* Supplier governance policy.
* Procurement standards.
* Governance committee minutes.
* Risk reports.
* Supplier registers.

# TP2 — Supplier Risk Assessment

## Objective

Determine whether supplier assessments reflect business criticality rather than applying a one-size-fits-all approach.

### Assessment Areas

Evaluate:

* Initial due diligence.
* Periodic reassessment.
* Security questionnaires.
* Independent assurance.
* Regulatory obligations.
* Financial resilience.
* Operational resilience.

### Sample Questions

| Question                                            | Score |
| --------------------------------------------------- | :---: |
| Supplier assessments are risk-based.                |       |
| Assessment frequency reflects supplier criticality. |       |
| Independent assurance reports are reviewed.         |       |
| High-risk findings require remediation plans.       |       |
| Risk acceptance decisions are documented.           |       |

# TP3 — Critical Supplier Management

## Objective

Evaluate governance of suppliers supporting critical business services.

### Assessment Areas

Assess:

* Critical supplier inventory.
* Dependency mapping.
* Executive ownership.
* Recovery commitments.
* Service level monitoring.
* Exit planning.

### Common Weaknesses

* No inventory of critical suppliers.
* Recovery obligations undefined.
* Supplier dependency maps incomplete.
* Exit strategies untested.

# TP5 — Software Supply Chain Security

## Objective

Evaluate the integrity and resilience of software acquisition, development, and deployment.

### Assessment Areas

Assess:

* Software Bill of Materials (SBOM).
* Dependency management.
* Open-source governance.
* Build pipeline security.
* Code signing.
* Artifact integrity.
* Package repository controls.

### Evidence Examples

* SBOM repositories.
* CI/CD security reports.
* Dependency scanning results.
* Code-signing policies.

### Maturity Indicators

**Level 1:** Software dependencies are largely unmanaged.

**Level 3:** Dependency inventories and vulnerability scanning are established.

**Level 5:** Software integrity is continuously verified across the entire software supply chain.
