# Appendix I.3 — Enterprise Cyber Resilience Maturity Model (ECRMM)

> *"Resilience is not measured by surviving one incident. It is measured by continuously improving the organization's ability to anticipate, withstand, recover, and adapt."*

---

# I.3 Enterprise Cyber Resilience Maturity Model (ECRMM)

## Purpose

The Enterprise Cyber Resilience Maturity Model (ECRMM) provides organizations with a structured framework for assessing, benchmarking, and continuously improving cyber resilience capabilities.

Unlike traditional security maturity models that focus primarily on technical controls, ECRMM evaluates the organization's ability to:

* Govern cyber risk.
* Protect critical business services.
* Detect and respond to cyber incidents.
* Recover from disruption.
* Adapt based on operational experience.

The model supports:

* Self-assessments.
* Executive reporting.
* Strategic planning.
* Investment prioritization.
* Regulatory readiness.
* Continuous improvement.

---

# I.3.1 Guiding Principles

The model is built on six principles:

1. **Business Outcomes Over Technical Outputs**
   Measure resilience in terms of business continuity, operational stability, and stakeholder confidence.

2. **Capability Over Compliance**
   Compliance is necessary but not sufficient. Mature organizations build enduring capabilities rather than simply satisfying audit requirements.

3. **Continuous Improvement**
   Maturity is iterative. Every exercise, incident, and assessment should contribute to measurable progress.

4. **Evidence-Based Assessment**
   Ratings should be supported by documented evidence, not subjective opinion.

5. **Executive Accountability**
   Cyber resilience is an enterprise responsibility shared by business and technology leaders.

6. **Risk-Informed Investment**
   Improvements should be prioritized based on business risk and strategic objectives.

---

# I.3.2 The Five Maturity Levels

| Level | Name       | Characteristics                                                                     |
| ----- | ---------- | ----------------------------------------------------------------------------------- |
| 1     | Initial    | Ad hoc, reactive, inconsistent practices.                                           |
| 2     | Developing | Basic processes established but inconsistently applied.                             |
| 3     | Defined    | Standardized processes implemented across the enterprise.                           |
| 4     | Managed    | Capabilities measured, governed, and continuously monitored.                        |
| 5     | Adaptive   | Continuous optimization driven by metrics, automation, and organizational learning. |

Progression between levels should require demonstrable evidence rather than aspirational statements.

---

# I.3.3 Capability Domains

The model evaluates ten enterprise capability domains:

| Domain                       | Purpose                                          |
| ---------------------------- | ------------------------------------------------ |
| Governance & Leadership      | Strategic oversight and accountability           |
| Risk Management              | Identification and treatment of cyber risk       |
| Identity & Access Management | Protection of identities and privileged access   |
| Detection & Monitoring       | Visibility and threat detection                  |
| Incident Response            | Coordinated response to cyber events             |
| Recovery & Continuity        | Restoration of critical services                 |
| Third-Party & Supply Chain   | Oversight of external dependencies               |
| Technology & Architecture    | Secure and resilient technology foundations      |
| People & Culture             | Awareness, training, and organizational behavior |
| Continuous Improvement       | Lessons learned, metrics, and capability growth  |

Each domain is assessed independently before determining an overall maturity profile.

---

# I.3.4 Assessment Methodology

Each capability domain is evaluated using four evidence categories:

| Category                  | Examples                                              |
| ------------------------- | ----------------------------------------------------- |
| Documentation             | Policies, standards, procedures                       |
| Implementation            | Technical controls, operational processes             |
| Operational Effectiveness | Metrics, testing results, audit findings              |
| Continuous Improvement    | Lessons learned, remediation tracking, trend analysis |

A capability cannot be rated solely on documented policies; evidence of implementation and effectiveness is required.

---

# I.3.5 Scoring Model

Each assessment criterion is scored on a scale from 0 to 5:

| Score | Description     |
| ----- | --------------- |
| 0     | Not Implemented |
| 1     | Initial         |
| 2     | Developing      |
| 3     | Defined         |
| 4     | Managed         |
| 5     | Adaptive        |

Domain scores are calculated by averaging the scores of their underlying assessment criteria.

---

# I.3.6 Assessment Process

The recommended assessment process consists of six stages:

1. Preparation
2. Evidence Collection
3. Interviews and Workshops
4. Capability Scoring
5. Executive Review
6. Improvement Planning

Cross-functional participation should include business leaders, technology teams, risk management, internal audit, and executive sponsors.

---

# I.3.7 Evidence Requirements

Every maturity rating should be supported by objective evidence, such as:

* Policies and standards.
* Architecture documentation.
* Exercise reports.
* Incident postmortems.
* Audit reports.
* Performance metrics.
* Training records.
* Recovery test results.

Statements without supporting evidence should not influence maturity ratings.

Organizations should visualize capability scores to identify strengths and weaknesses.

Example:

| Capability Domain      | Score |
| ---------------------- | ----: |
| Governance             |   4.3 |
| Risk Management        |   3.9 |
| Identity               |   3.2 |
| Detection              |   3.8 |
| Incident Response      |   3.3 |
| Recovery               |   2.9 |
| Third-Party Risk       |   2.8 |
| Architecture           |   3.6 |
| Culture                |   3.5 |
| Continuous Improvement |   3.1 |

Rather than averaging these into a single number, organizations should treat the profile as a portfolio of capabilities requiring different investment strategies.

---

# I.3.10 Improvement Roadmap

For each domain:

1. Identify current maturity.
2. Define target maturity.
3. Prioritize gaps based on business risk.
4. Assign accountable owners.
5. Allocate funding.
6. Track progress quarterly.

Improvement roadmaps should generally cover a 2–3 year horizon.

---

# I.3.11 Governance Integration

Assessment results should feed into:

* Enterprise risk management.
* Strategic planning.
* Annual budgeting.
* Board reporting.
* Internal audit.
* Regulatory engagement.

This ensures that maturity assessments influence organizational decisions rather than becoming isolated compliance exercises.

---

# I.3.12 Success Criteria

An organization demonstrates increasing maturity when it can show:

* Improved recovery performance.
* Faster detection and response.
* Reduced recurrence of significant findings.
* Increased executive engagement.
* Better alignment between cybersecurity investments and business outcomes.
* Sustained improvement across multiple assessment cycles.
