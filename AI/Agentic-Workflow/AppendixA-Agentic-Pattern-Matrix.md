# Appendix A: The Agentic Pattern Matrix

## Purpose

A decision reference for choosing an agentic pattern for a new problem, rather than defaulting to whichever pattern you built most recently — which is a more common failure mode than it sounds. Once you've spent real time building, say, a Reflective Plan-and-Execute agent, there's a natural pull toward reaching for that same heavyweight pattern on the next problem too, simply because it's fresh and familiar, even when the next problem is a two-step classification task that never needed any of that machinery. Each pattern in the matrix below has a genuinely distinct cost, reliability, and latency profile, and picking the wrong one — in either direction — is a common source of over-engineered systems (a Reflective Multi-Agent pipeline built to answer a question a single prompt could have handled) and under-engineered ones (a bare ReAct loop asked to carry a long, drift-prone, multi-day task it was never designed to hold together). This Appendix exists to make that choice deliberate, evidence-based, and revisitable, rather than a one-time architectural instinct you never reconsider.

## The Matrix

| Pattern | Best For | Reliability Profile | Latency | Cost | Series Reference |
|---|---|---|---|---|---|
| **Simple Chain** (one model call) | Fixed-shape tasks: classification, extraction, single-step transformation | Highest — no branching to go wrong | Lowest | Lowest | Not covered as its own Part — the "do you even need an agent?" baseline |
| **ReAct** (Reason-Act loop) | Task-based, single-session goals, small unknown step count | Medium — can loop or pick wrong tools; bounded by step ceiling | Medium — scales with steps taken | Medium — pay per step | Part 1, Part 2 |
| **Plan-and-Execute** | Long-running, multi-step goals where upfront decomposition reduces drift | Medium-High — plan reduces drift but is "sticky" until re-planned | Higher — planning adds latency upfront | Higher — planning call + N execution calls | Part 4 |
| **Reflective (Generation-Critique-Refinement)** | Layered on top of any pattern, when correctness/safety matters more than speed | Highest achievable with LLMs | Highest — doubles+ calls per gated step | Highest — same multiplier as latency | Part 5 |
| **Multi-Agent** (Planner delegates to specialized agents) | Specialized-role tasks where a single prompt can't hold all necessary context/expertise | Variable — depends on inter-agent protocol design; new failure mode (miscommunication) | Highest — multiple independent agent loops | Highest — N agents each with own call budget | Part 9a, Part 9b |

A row deliberately absent from this version of the table: **Deterministic Workflow (n8n)**. It's tempting to list it alongside the others as if it were one more point on the same spectrum, but that would misrepresent what it actually is. Every other row in this table is a way of letting a model decide something at runtime — the rows differ in *how much* runtime decision-making they involve and *how it's structured*, but they're all, fundamentally, reasoning patterns. A deterministic n8n workflow involves no runtime model decision-making at all, by design — it's not a weaker or cheaper reasoning pattern, it's the *absence* of one. Ranking it as a row alongside ReAct or Multi-Agent invites the wrong comparison — "which pattern is best" — when the real first question, addressed directly in the decision heuristic below, is "does this problem need a reasoning pattern in the first place." Keep n8n (or any deterministic workflow tool) as the answer to that first question, not as a competing entry in a table of reasoning patterns.

A second thing worth reading directly out of the table before moving to the heuristic: look at how consistently latency and cost move together, row by row, while reliability does *not* move in lockstep with them. Reflective is the most expensive pattern per gated step and also the most reliable — that correlation holds. But Multi-Agent is *also* among the most expensive patterns, and its reliability is explicitly marked "Variable," not "Highest." That's not a table error; it's the single most important nuance in the whole matrix, and it's worth restating because it cuts against a natural intuition: spending more (latency, cost) does not uniformly buy you more reliability. Reflection spends more and reliably gets more, because the mechanism — a second, adversarial reasoning pass — has a clear, well-understood relationship to catching a first pass's errors. Multi-Agent spends more and gets a *new kind of risk* in exchange — the inter-agent handoff itself is a place things can go wrong that simply doesn't exist in a single-agent system, no matter how well-designed the contracts are (Part 9a, section 3's whole argument for Zod-validated handoffs exists precisely because that new risk is real, not hypothetical). Treat "more expensive" and "more reliable" as two independent axes when reading this table, not one.

## Decision Heuristic (apply in this order)

1. **Can this be a fixed sequence of steps a human could flowchart without knowing what an LLM "thinks"?** → Deterministic workflow (n8n, or equivalent). Do not reach for an agent. This is the same test Part 6, section 1 introduced for deciding what belongs in n8n versus LangGraph, and it's worth applying it *first*, before any of the rows in the table above even enter consideration — every step down this list assumes you've already confirmed genuine model judgment is required somewhere in the task, and this first check is what confirms that.

2. **Is the task small enough to complete in a handful of steps, where the model deciding "what's next" each step is genuinely useful?** → ReAct (Parts 1-2). This is the right default once step 1 has confirmed real judgment is needed — resist the pull toward a heavier pattern just because it exists in your toolbox. A task that can be handled well by ReAct and is instead routed through Plan-and-Execute or Multi-Agent isn't more robust for it; it's just slower and more expensive for no reliability gain, per the independence of the two axes just discussed.

3. **Does the task span enough steps or time that re-deriving strategy every step causes drift or excessive cost?** → Plan-and-Execute (Part 4). Notice this question is specifically about *drift and re-derivation cost*, not simply "is this a long task." A long task made of many nearly-identical, independent steps (processing a big batch of similar tickets, say) may not need explicit planning at all — ReAct in a loop, invoked once per item, might be entirely sufficient and considerably cheaper. Plan-and-Execute earns its upfront planning latency specifically when the steps are *interdependent* and *coordinated toward one goal*, which is where re-deriving the overall strategy at every step (Part 4, section 1's stated failure mode) actually becomes a real cost rather than a theoretical one.

4. **Is being wrong expensive — an irreversible action, a compliance-sensitive output, a user-facing claim?** → Add Reflection (Part 5) on top of whichever pattern you chose in steps 2 or 3. The word "add" matters here: Reflection is explicitly not its own row you pick *instead of* ReAct or Plan-and-Execute, it's a layer you apply *on top of* one of them, exactly as Part 5 built it (wrapping the existing Reason/Act loop with Critique/Refine) and exactly as Part 9b built it again (the Coder/Reviewer pair being, structurally, Reflection's Generator/Critic pattern promoted to two separate agents). Reflection is best thought of as orthogonal to the choice of base pattern, not competing with it — you can and often should combine it with whichever pattern step 2 or 3 already selected.

5. **Does the task require genuinely different expertise or context per sub-problem that a single prompt can't hold well — and do you have measured evidence of that, not just a hunch?** → Consider Multi-Agent (Parts 9a-9b). The qualifier at the end of this question is doing the real work, and it's worth treating as a hard gate rather than a soft suggestion: "consider," not "adopt." Everything in the next section expands on exactly what kind of evidence should clear that gate, because this is the step most teams get wrong — reaching for Multi-Agent based on a task *sounding* like it needs specialized roles, rather than having actually measured that a single agent is failing specifically because of role-mixing.

## The Multi-Agent Question, Revisited

*(A note on this section's history, left in deliberately rather than silently edited away: this Appendix was originally written before Parts 9a and 9b existed, back when Multi-Agent was still a hypothetical follow-up pattern rather than a built reference implementation. The reasoning below has been brought forward to reflect that it now exists and has been built — but the underlying caution is unchanged, and if anything, having actually built it in Parts 9a-9b makes the caution more concrete rather than less.)*

Earlier in this series, Part 5's Critique node was deliberately framed as the simplest possible instance of "multiple reasoning roles cooperating" — one Generator role, one Critic role, communicating through shared state within a single graph, a single process, a single set of credentials. That framing was not just a pedagogical simplification; it was making an implicit argument that most of what a "second opinion" or "specialized role" buys you can often be achieved *within* a single agent's graph, at a fraction of the cost and complexity of standing up genuinely separate agents — and that argument held for four full Parts (1 through 8) before this series ever introduced a second agent.

A full Multi-Agent architecture, as Parts 9a and 9b actually built it, is the natural next escalation once you have concrete evidence — from Part 7's traces and Appendix B's evaluation framework, specifically, not from intuition alone — that a single agent's context or role-mixing is the actual bottleneck. Part 9a, section 1 named this precisely: a system prompt being asked to hold two genuinely different skills simultaneously (writing code, adversarially reviewing it), where traces showed that mixing was producing worse outcomes than the two jobs split apart. That's the bar. Notice what clearing it required in practice: not "code review sounds useful," not "specialized agents are the more sophisticated architecture," but a specific, falsifiable, measured finding about where an already-built, already-instrumented single-agent system was actually struggling.

**The staff-engineer caution on Multi-Agent specifically, now grounded in a system that's actually been built rather than a hypothetical one:** it is the most expensive pattern in this matrix along every dimension — latency, cost, and, counterintuitively, reliability, because inter-agent handoffs are a genuinely new class of failure that does not exist in a single-agent system at all. Part 9a's entire section 3 (shared Zod contracts between agents) and its entire section 4 (tool- and model-level role isolation) exist because that new failure class is real: a raw-string handoff between two agents can silently propagate a misunderstanding in a way that has no equivalent inside one graph's shared, typed state object. Having now built the mitigation doesn't eliminate the risk category, it just gives you a well-tested way to manage it — the same way Part 2's `ToolResult` contract doesn't eliminate the possibility of tool failure, it just makes failure something the system can reason about instead of crash on.

Reach for Multi-Agent only when you have a measured, specific reason Plan-and-Execute-with-Reflection isn't sufficient — and if you've reached that point, Parts 9a-9b are no longer a hypothetical sketch of what that might look like, they're a working reference implementation you can adapt directly: the contract-first handoff discipline, the tool/model isolation per role, the orchestrator-as-sole-integrator pattern, the shared correlation ID threading, and the escalation path reusing the exact same n8n action layer as every single-agent side effect in this series. That's the concrete payoff of this Appendix's history being preserved rather than quietly rewritten: the caution that stopped this series from building Multi-Agent for eight full Parts is the same caution that should stop you from reaching for it prematurely in your own systems — right up until you have the kind of evidence that made it the right call here.

## Applying the Matrix: A Worked Example

To make the heuristic concrete rather than purely abstract, walk one hypothetical task through all five steps, the way you'd actually reason about a new problem in practice: *"When a support ticket comes in, draft a reply, and if it involves a refund, send it only after a teammate approves."*

- **Step 1:** Is this a fixed flowchart? Partially — "route the ticket, check if it mentions a refund, wait for approval, send" is flowchart-able, but "draft a reply" requires understanding the ticket's actual content, which a fixed sequence of steps can't do. This is the signal that *part* of the task is n8n's job and *part* of it needs reasoning — exactly the hybrid split Part 6 built, not a reason to abandon deterministic workflow entirely.
- **Step 2:** Is the reasoning part small enough for plain ReAct? Likely yes — drafting one reply based on one ticket's content, with maybe a knowledge-base lookup (Part 3), is a small, single-session task with no real drift risk.
- **Step 3:** Does it need Plan-and-Execute? Probably not — there's no long chain of interdependent steps here that would benefit from upfront decomposition.
- **Step 4:** Is being wrong expensive? Yes, clearly — a wrong or inappropriate reply to a customer, or worse, an unauthorized refund, is exactly the kind of irreversible, user-facing, compliance-adjacent output Part 5's Reflection layer is for. This is where the design should land: ReAct plus Reflection, with the refund-approval step handled by the confirmation-gated tool pattern from Part 2 and the actual approval routed through n8n and a human, per Part 5's HITL escalation.
- **Step 5:** Does it need Multi-Agent? Nothing here suggests a single prompt can't hold "understand the ticket and draft a reply" as one coherent role — there's no evidence, and no obvious reason to expect, that role-mixing is a bottleneck. Skip it, at least until real traces say otherwise.

The resulting architecture — deterministic routing (n8n) around a ReAct-plus-Reflection core (Parts 1, 2, 5), with a confirmation-gated action (Part 2) executed through the n8n action layer (Part 6) and traced end-to-end (Part 7) — uses exactly four of this series' patterns and deliberately skips Plan-and-Execute and Multi-Agent, not because those patterns are worse, but because nothing about this specific task justified their cost. That's the matrix doing its actual job: not telling you which pattern is best in the abstract, but telling you which patterns this specific problem has actually earned.
