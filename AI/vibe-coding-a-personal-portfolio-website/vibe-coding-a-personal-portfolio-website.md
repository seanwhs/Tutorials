# The Architecture of Intent: Why Engineering Precedes the Prompt
## A Practical Guide to Vibe Coding a React + Sanity Portfolio for Freelance Consulting, Training, and Development with OpenCode, GSD, and Antigravity Skills

The most common failure in AI-assisted development is not a lack of coding skill.

It is a failure of system definition.

Developers frequently approach AI as if it were an extraordinarily fast software contractor:

"Build me a portfolio."  
"Create a dashboard."  
"Generate a SaaS application."  
"Make me a consultant website."

The AI complies.  
Pages appear.  
Components render.  
APIs connect.  
Tests may even pass.

Yet despite the apparent success, the resulting system often feels generic, fragile, or directionless.

The code is correct.  
The architecture is absent.

The AI generated exactly what was requested.  
The problem is that nobody spent enough time defining what should exist in the first place.

This is one of the central paradoxes of AI-native development:  
**The easier code becomes to generate, the more important architectural thinking becomes.**

In traditional software development, implementation effort naturally slowed people down. Developers were forced to think because changing direction was expensive.

AI removes much of that friction.

Ideas can become applications in minutes.

But speed introduces a new danger:  
**Systems can now be built faster than they can be understood.**

The result is a new class of failure:

Fast generation.  
Slow understanding.

The application exists.  
The architecture does not.

***

## The Shift from Generation to Architecture

Many discussions about AI focus on generation.

How quickly can code be produced?  
How many files can be created?  
How many features can be scaffolded?

These questions are increasingly becoming irrelevant.

Generation is becoming abundant.  
Architecture remains scarce.

The bottleneck is no longer:

Can we build it?

The bottleneck is:

Should it exist?  
What should it become?  
How should it evolve?  
What responsibilities belong where?

AI has dramatically reduced the cost of implementation.  
It has not reduced the cost of making good decisions.  
In many ways, it has increased it.  
Because poor decisions can now be amplified at unprecedented speed.

***

## The Six Foundational Conversations (Revised for Freelance)

Before the first line of code is generated in OpenCode, six architectural conversations must occur.

These conversations define **System Intent**.

System Intent is the complete description of what a system exists to accomplish, who it serves, how it creates value, and how it should evolve.

The AI does not invent this.  
The engineer does.  
The AI merely operationalizes it.

### I. Defining the Job to Be Done

Every application exists to perform a job.  
The question is: **What job?**

For a freelance consultant, trainer, and developer, the job is:

**Freelance Engineering Commerce Platform**  
Optimized for: selling consulting, training, and development services  
Primary assets: service offerings, case studies, curriculum, expertise, testimonials, booking, and contact

This is not a career platform.  
This is not a knowledge platform.  
This is not a teaching platform.

It is a **hybrid commerce system** that serves three audiences with three different conversion paths.

The chosen job determines every downstream architectural decision.

### II. Identifying the Primary Users

Every system serves someone.  
Trying to serve everyone typically results in serving nobody particularly well.

For your freelance pivot, you have three primary audiences:

**Hiring Companies / Startup CEOs** (consulting)  
Ask: *Can you solve my technical problem and deliver architecture?*  
Seek: case studies, technical depth, system design, ADRs, outcomes

**Enterprise / Organization Leaders** (training)  
Ask: *Can you teach my team and upgrade their skills?*  
Seek: curriculum, learning paths, teaching methodology, certifications, testimonials

**Product Teams / Engineering Managers** (development)  
Ask: *Can you build my system with modern frameworks and AI integration?*  
Seek: project portfolio, technical stack, delivery timeline, code quality

Each audience asks different questions.  
Each audience needs different navigation paths.  
Each audience converts at different points in the site.

The primary audience mix determines:

Navigation  
Service pages  
Content structure  
Information hierarchy  
Feature priority  
Booking and contact flows

Architecture begins with audience selection.

For a freelance consultant, trainer, and developer, the product is not just projects.

## The Practical Build Sequence for React + Sanity (Freelance Edition)

If you are vibe coding a freelance consultant, trainer, and developer portfolio with React and Sanity in OpenCode, use this sequence:

1. **Define the job of the site** (freelance commerce platform for consulting, training, development) using `@brainstorming`.
2. **Define the three primary audiences** (companies, leaders, teams) using `@concise-planning`.
3. **Decide whether the product is a portfolio or a freelance business operating system** (definitely the latter) using `@brainstorming`.
4. **Identify the content types that deserve first-class status**: Services, Case Studies, Curriculum, Testimonials, Packages, Booking using `@frontend-design`.
5. **Model those types in Sanity** with commerce fields (pricing, packages, booking) using `@react-best-practices` and `@tailwind-patterns` as context.
6. **Map the relationships between services, case studies, and testimonials** using `@brainstorming`.
7. **Build the React interface around the model** with service pages, booking forms, and case study anatomy using `@react-patterns`.
8. **Use AI to accelerate implementation only after the system intent is clear**, guided by GSD phases.
9. **Review whether the site communicates reasoning, outcomes, and delivery capability**, using `@lint-and-validate`.
10. **Iterate on service structure and pricing before aesthetics**, using `@systematic-debugging` for issues.

This sequence keeps implementation subordinate to architecture.  
That is what prevents AI from producing a generic services site that fails to express your expertise and business model.

## GSD Workflow: The Six Phases

GSD structures development into six phases:

1. **Project initialization**  
   GSD asks for idea, constraints, tech stack, and creates `PROJECT.md`, `ROADMAP.md`.

2. **Discuss phase**  
   Implementation details are defined, creating `CONTEXT.md`.

3. **Plan phase**  
   Parallel research and planning, with XML-structured task units.

4. **Execution phase**  
   Dependency-based wave parallel execution, commit per task, verification.

5. **Verification phase**  
   Automated tests and user confirmation; auto-fix plan on failure.

6. **Iteration and milestone completion**  
   Repeat phases, then release tagging. [hi.news.hada](https://hi.news.hada.io/topic?id=27622)

In OpenCode, you run GSD commands like:

- `/gsd:new-project`  
- `/gsd:plan-phase`  
- `/gsd:execute-phase`  
- `/gsd:verify-work`  
- `/gsd:ui-phase`  
- `/gsd:map-codebase` [hi.news.hada](https://hi.news.hada.io/topic?id=27622)

```text
Act as a senior software architect and GSD coach.

Use @brainstorming to help me define the system intent for a freelance consultant, trainer, and developer platform built with React and Sanity, vibe coded in OpenCode.

- The unique value proposition (consulting and engineering anatomy vs. generic service cards)
- The expected evolution path (blog, courses, labs, research, booking system, pricing tiers)

After I answer, summarize the system intent in a concise architecture brief.
Create PROJECT.md and ROADMAP.md according to GSD initialization.
Do not generate code yet.
```

### 2. Discuss Phase Prompt (Freelance Edition)

```text
Act as a GSD system engineer.

Use @concise-planning to define the implementation context for my React + Sanity freelance platform.

Based on the system intent from PROJECT.md:
- Create CONTEXT.md with implementation details
- Clarify the folder structure for React components, Sanity schemas, and pages
- Define the editorial workflow in Sanity Studio for services, case studies, and curriculum
- Specify the data model and relationships between services, case studies, testimonials, and packages
- Add commerce fields: pricing, packages, booking slots, contact flows

Prefer architecture over visual design.  
Do not write implementation code unless I ask for it.
```

### 3. Plan Phase Prompt (Freelance Edition)

```text
Act as a GSD planner.

Use @brainstorming and @concise-planning to produce the plan phase for my React + Sanity freelance platform.

Generate:
- XML-structured task units for each component and schema (Services, Case Studies, Curriculum, Testimonials, Packages, Booking)
- Parallel research tasks for Sanity content modeling best practices for commerce
- A roadmap of milestones:
  • M1: System intent and content model for consulting, training, development
  • M2: Sanity schemas with commerce fields
  • M3: React service components and case study anatomy
  • M4: Pages: Services, Case Studies, Curriculum, Testimonials, Booking, About
  • M5: Testing and deployment

For each task unit:
- Define the goal
- List the deliverables
- Identify risks
- State acceptance criteria

Do not collapse phases into one task.  
Keep the work small, explicit, and reviewable.
```

### 4. Execution Phase Prompt (Freelance Edition)

```text
Act as a GSD execution agent.

Use @react-best-practices and @tailwind-patterns to implement the first wave of tasks from the plan.

Tasks:
- Create the Sanity schema for Service Offering (consulting, training, development)
- Create the Sanity schema for Case Study with engineering anatomy
- Create the Sanity schema for Curriculum with modules and labs
- Create the Sanity schema for Testimonial
- Create the Sanity schema for Package with pricing tiers
- Create the Sanity schema for Booking Slot
- Scaffold the React component structure (ServiceCard, CaseStudyAnatomy, CurriculumList, TestimonialGrid, Package pricing, Booking form)

Rules:
- Use the Sanity content model as the source of truth.
- Keep components small and composable.
- Preserve separation between content, layout, and presentation.
- Avoid hard-coded service sections unless they are justified by the system intent.
- Prioritize clarity, maintainability, and commerce discoverability.
- Add commerce fields: pricing, packages, booking, contact flows.

After each task, commit with git using @git-pushing.
Verify the implementation using @lint-and-validate.
```

### 5. UI Phase Prompt (Freelance Edition)

```text
Act as a GSD UI designer.

Use @frontend-design and @tailwind-patterns to implement the UI layer for the freelance platform.

Tasks:
- Build the Home page with hero section, three service pillars (consulting, training, development), featured case studies, and latest curriculum
- Build the Services page with filtering by consulting/training/development and service anatomy cards
- Build the Case Studies page with engineering anatomy and outcomes
- Build the Curriculum page with course list, modules, and tags
- Build the Testimonials page with client quotes and outcomes
- Build the Packages page with pricing tiers and features
- Build the Booking page with form, calendar, and contact flows
- Build the About page with expertise, background, and credentials

Rules:
- Use Tailwind for styling, not ad-hoc utilities.
- Ensure responsive layouts for mobile, tablet, and desktop.
- Implement proper routing and navigation.
- Keep the design system consistent across pages.
- Emphasize commerce and booking flows.

Commit each page with git using @git-pushing.  
Validate with @lint-and-validate.
```

### 6. Verification Phase Prompt (Freelance Edition)

```text
Act as a GSD verifier.

Use @lint-and-validate and @systematic-debugging to verify the freelance platform implementation.

Tasks:
- Run linter checks on all React components and Sanity schemas
- Validate that all components use predefined Tailwind styles
- Confirm that routing and navigation work correctly
- Check that Sanity Studio loads and schemas are registered
- Run basic tests for page rendering
- Verify that booking forms and contact flows work
- Test pricing and package displays

If failures occur:
- Generate an auto-fix plan
- Execute fixes using @systematic-debugging
- Re-run verification

Produce a verification report with pass/fail for each task.
```

### 7. GSD Review Prompt (Freelance Edition)

```text
Act as a GSD reviewer.

When you run `/gsd:review` (or `/gsd_review` in OpenCode), the workflow:
1. Detects which CLIs are available (opencode and claude)
2. Gathers the phase context (PROJECT.md, ROADMAP.md, PLAN.md, CONTEXT.md)
3. Builds a structured review prompt
4. Invokes all 6 reviewers in parallel

Use this to review the current milestone of my React + Sanity freelance platform.
Evaluate:
- Adherence to system intent (freelance commerce platform for consulting, training, development)
- Quality of content model for services, case studies, curriculum
- Component structure and reusability
- UI consistency and responsiveness
- Editorial workflow in Sanity
- Commerce and booking flows
```
