# Part 7 — Generating Professional PDF Grading Reports

Up to this point, Markly has become a capable AI grading assistant.

A teacher can:

* Upload a PDF assignment
* Upload a Microsoft Word document
* Upload a scanned worksheet
* Upload a handwritten assignment
* Select a subject
* Receive AI-generated feedback

That's already useful, but there is one problem.

The feedback only exists inside the browser.

Once the teacher refreshes the page or closes the browser, the feedback disappears.

In a real classroom, teachers often need to:

* Print grading reports
* Email feedback to students
* Archive assessments
* Upload reports to a Learning Management System (LMS)

Instead of copying and pasting the AI's response into Microsoft Word, we'll automate the process.

By the end of this chapter, Markly will generate professional PDF grading reports that teachers can download with a single click.

# Why Generate PDFs?

PDF (Portable Document Format) has become the standard format for sharing documents.

Unlike Word documents:

* PDFs look the same on every computer.
* Fonts remain consistent.
* Layouts don't change.
* Documents are difficult to accidentally edit.
* PDFs are ideal for printing.

This makes them perfect for assessment reports.

Imagine receiving something like this after grading an assignment.

```text
--------------------------------------------------

             MARKLY GRADING REPORT

Student:
John Tan

Subject:
Mathematics

Assignment:
Algebra Worksheet

----------------------------------------------

• Working is clearly shown.

----------------------------------------------

• Incorrect sign when moving constants.

• Check calculations more carefully.

----------------------------------------------

Suggestions

• Practice solving linear equations involving
negative numbers.

----------------------------------------------

Final Grade

8 / 10

Generated by Markly

--------------------------------------------------
```

Instead of displaying feedback only in the browser, we can generate this document automatically.

# Introducing ReportLab

To generate PDFs, we'll use **ReportLab**.

ReportLab is one of the most widely used PDF libraries in Python.

Unlike Microsoft Word, ReportLab doesn't use a visual editor.

Instead, we create the document entirely using Python code.

Think of it as drawing on a blank sheet of paper.

```text
Blank PDF Page

↓

Draw Title

↓

Draw Student Name

↓

Draw Feedback

↓

Save PDF
```

Everything is positioned using coordinates.

Although this may sound complicated, ReportLab provides high-level tools that make document generation much easier.

# Our First PDF

Open

```text
report.py
```

Let's generate a very simple PDF.

```python
import io

from reportlab.pdfgen import canvas


def create_pdf():

    buffer = io.BytesIO()

    pdf = canvas.Canvas(buffer)

    pdf.drawString(
        100,
        750,
        "Hello from Markly!"
    )

    pdf.save()

    buffer.seek(0)

    return buffer
```

Let's examine each step.

---

# Understanding BytesIO (Again)

Earlier, we used `BytesIO` to read uploaded Word documents.

Now we're using it for the opposite purpose.

Instead of reading,

we're writing.

```text
ReportLab

↓

Creates PDF

↓

Writes into BytesIO

↓

Return BytesIO

↓

Download
```

Nothing is written to disk.

The PDF exists entirely in memory.

This makes the application much faster and easier to deploy.

---

# Understanding Coordinates

Unlike Word,

a PDF page has coordinates.

The origin is located at the bottom-left corner.

```text
(0,800)

+-----------------------+

|                       |

|                       |

|                       |

|                       |

+-----------------------+

(0,0)
```

When we write

```python
pdf.drawString(
    100,
    750,
    "Hello"
)
```

we're saying

> Draw the text "Hello" at position (100, 750).

# Creating a Real Report

Let's replace the simple example with something more useful.

```python
from reportlab.lib.units import inch

from reportlab.pdfgen import canvas

import io


def create_pdf_report(

    student,

    subject,

    filename,

    feedback

):

    buffer = io.BytesIO()

    pdf = canvas.Canvas(buffer)

    y = 780

    pdf.setFont(
        "Helvetica-Bold",
        18
    )

    pdf.drawString(
        72,
        y,
        "Markly Grading Report"
    )

    y -= 40

    pdf.setFont(
        "Helvetica",
        11
    )

    pdf.drawString(
        72,
        y,
        f"Student: {student}"
    )

    y -= 20

    pdf.drawString(
        72,
        y,
        f"Subject: {subject}"
    )

    y -= 20

    pdf.drawString(
        72,
        y,
        f"Assignment: {filename}"
    )

    pdf.save()

    buffer.seek(0)

    return buffer
```

Run the function again.

The report now includes useful metadata.

# The Problem with drawString()

There's one major limitation.

Suppose the AI produces this paragraph.

```text
The student demonstrates a strong understanding
of algebraic manipulation but occasionally makes
errors when working with negative numbers...
```

If we write

```python
pdf.drawString(
    72,
    y,
    feedback
)
```

everything appears on **one extremely long line**.

The text runs off the page.

Clearly we need a better solution.

---

# Introducing Platypus

Fortunately,

ReportLab includes a higher-level framework called **Platypus**.

Instead of drawing every line manually,

Platypus understands:

* paragraphs
* headings
* spacing
* line wrapping
* page breaks

Think of it as the "word processor" layer built on top of ReportLab.

Instead of this

```text
Draw String

Draw String

Draw String

Draw String
```

we simply build a document made of paragraphs.

---

# Creating a Better Report

Replace the earlier implementation with this one.

```python
from reportlab.lib.styles import getSampleStyleSheet

from reportlab.platypus import (

    SimpleDocTemplate,

    Paragraph,

    Spacer

)

import io


def create_pdf_report(

    student,

    subject,

    filename,

    feedback

):

    buffer = io.BytesIO()

    document = SimpleDocTemplate(buffer)

    styles = getSampleStyleSheet()

    story = []

    story.append(

        Paragraph(

            "<b>Markly Grading Report</b>",

            styles["Title"]

        )

    )

    story.append(
        Spacer(1,12)
    )

    story.append(

        Paragraph(

            f"<b>Student:</b> {student}",

            styles["BodyText"]

        )

    )

    story.append(

        Paragraph(

            f"<b>Subject:</b> {subject}",

            styles["BodyText"]

        )

    )

    story.append(

        Paragraph(

            f"<b>Assignment:</b> {filename}",

            styles["BodyText"]

        )

    )

    story.append(
        Spacer(1,24)
    )

    story.append(

        Paragraph(

            feedback.replace(
                "\n",
                "<br/>"
            ),

            styles["BodyText"]

        )

    )

    document.build(
        story
    )

    buffer.seek(0)

    return buffer
```

This version is much easier to extend.

Notice that we no longer worry about page coordinates.

Platypus automatically handles:

* line wrapping
* spacing
* margins
* page layout
* page breaks

This makes it the preferred approach for generating reports with substantial amounts of text.

# Downloading the PDF in Panel

Now that we can generate a report, we need a way for teachers to download it.

Panel provides a convenient `FileDownload` widget.

In `app.py`, create a download button.

```python
download = pn.widgets.FileDownload(

    label="Download Report",

    button_type="success",

    filename="grading_report.pdf"
)
```

Unlike a normal button, `FileDownload` is designed specifically for serving files to the browser.

At the moment, however, it doesn't know *what* to download.

We'll connect it to our report generator after the grading completes.

At this point, Markly is already a practical application. A teacher can upload assignments, receive AI-generated feedback tailored to the selected subject, and export the results as a professional PDF report.

However, there's still one area where the user experience can be improved. The application currently performs every task synchronously, meaning the interface appears to "freeze" while the AI is generating its response. Additionally, teachers receive very little indication of what is happening behind the scenes.

In the next instalment, we'll focus on **building a polished user experience**. We'll add progress indicators, status messages, error handling, loading spinners, and responsive controls so that Markly feels like a professional desktop application rather than a simple prototype. These improvements won't change the AI itself, but they'll make the application significantly more pleasant and reliable to use.
