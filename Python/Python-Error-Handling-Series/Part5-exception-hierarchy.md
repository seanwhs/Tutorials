# Reverse Engineering Python's Exception Hierarchy: Why `except Exception` Doesn't Catch Everything

> *"Most developers see exceptions as a list of error types.*
>
> *Professional engineers see them as a taxonomy of failure."*

---

# Introduction

Consider this code:

```python
try:
    dangerous()

except Exception:
    print("handled")
```

Many developers believe:

> "This catches everything."

It doesn't.

For example:

```python
try:
    raise KeyboardInterrupt()

except Exception:
    print("caught")
```

Output:

```text
KeyboardInterrupt
```

No catch.

Why?

Because Python's exception hierarchy wasn't designed as a list of errors.

It was designed as a model of:

> **which failures applications should recover from, and which failures applications should never interfere with.**

Understanding this hierarchy is one of the biggest jumps from beginner Python to professional Python.

---


```python

    recover()
```

* bugs,
* user interrupts,
* program termination,
* interpreter shutdown.

That would be disastrous.



## Application Failures

```text

```

from:

## Runtime Control Signals

```text
BaseException subclasses
```

# Chapter 62 ÔÇö The Professional Catching Strategy

Catch:

```text
Specific Exception
        |
        V
Exception Family
        |
        V
Exception
```

```text
BaseException
```

---

Example:

```python

    process()

    recover()

    retry()

    raise
```
