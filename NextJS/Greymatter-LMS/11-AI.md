## 2. Replacing the Grading Worker's fake score with real AI

Back in Part 7, our Grading Worker returned `Math.floor(Math.random() * 41) + 60` instead of an actual grade [3]. Let's fix that first, since it's the worker we already know end-to-end.

```typescript
// workers/grading-worker/index.ts (real AI grading)
import express from "express";
import OpenAI from "openai";
import { verifySignature, signPayload } from "../../packages/workers/sign";
import type { WorkerInput, WorkerOutput } from "../../packages/workers/types";

const app = express();
app.use(express.json());
const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });
const SECRET = process.env.WORKER_SIGNING_SECRET!;

app.post("/api/grading-worker", async (req, res) => {
  const signature = req.header("x-signature") ?? "";
  const input: WorkerInput = req.body;

  if (!verifySignature(input, signature, SECRET)) {
    return res.status(401).json({ error: "Invalid signature" });
  }

  const completion = await openai.chat.completions.create({
    model: "gpt-4o-mini",
    messages: [
      { role: "system", content: "Grade this student submission from 0-100 and give one sentence of feedback. Respond as JSON: { score: number, feedback: string }." },
      { role: "user", content: input.submission.content },
    ],
    response_format: { type: "json_object" },
  });

  const graded = JSON.parse(completion.choices[0].message.content!);

  const output: WorkerOutput = {
    workerName: "grading-worker",
    resultType: "grading",
    data: { score: graded.score, feedback: graded.feedback },
    success: true,
  };

  res.setHeader("x-signature", signPayload(output, SECRET));
  res.json(output);
});

app.listen(4000, () => console.log("Grading Worker (real AI) listening on :4000"));
```

Nothing else changes — the signature verification, the `WorkerInput`/`WorkerOutput` shapes, and the Sanity registry document from Part 6 all stay exactly as they were [4].

**✅ Checkpoint:** Submit an assignment with genuinely different quality content (a thoughtful answer vs. a one-word answer) and confirm the score and feedback text actually reflect the difference — not a random number. Then check `workflow_logs` from Part 10 and confirm the run still appears in your execution timeline, traced end-to-end as before [11].

## 4. Real AI tutoring — upgrading the intervention chain

Part 8's `student-struggling` function used a single hardcoded sentence as a stand-in for tutor intervention [2]. Now that the pipeline and chaining logic are proven, let's give it real intelligence:

```typescript
// infra/inngest/functions/studentStruggling.ts (real AI tutor intervention)
import OpenAI from "openai";

const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

export const studentStruggling = inngest.createFunction(
  { id: "student-struggling" },
  { event: "student.struggling" },
  async ({ event, step }) => {
    if (!event.data.submissionId || !event.data.studentId) {
      throw new Error("Rejected: student.struggling event missing required internal context");
    }

    const intervention = await step.run("tutor-intervention", async () => {
      const completion = await openai.chat.completions.create({
        model: "gpt-4o-mini",
        messages: [
          { role: "system", content: "A student is struggling (score below 70). Write a short, encouraging, specific tip to help them improve." },
          { role: "user", content: `Student's score: ${event.data.score}` },
        ],
      });

      return { message: completion.choices[0].message.content };
    });

    await step.run("emit-practice-assigned", async () => {
      await inngest.send({
        name: "practice.assigned",
        data: { studentId: event.data.studentId, reason: "struggling", intervention, traceId: event.data.traceId },
      });
    });

    return intervention;
  }
);
```

Note this is a change *inside* an existing Inngest function, not a new worker — tutor intervention lives in the Orchestration Layer's own logic in this design, rather than as a separate registered worker, which is a deliberate simplification worth noting as you build.

**✅ Checkpoint:** Force a low score again (as in Part 8's conditional-branch checkpoint [2]) and confirm the `tutor-intervention` step now returns a genuinely written, context-aware message — then check its `traceId` in the Part 10 timeline view to confirm it's still linked to the original submission [11].

## 5. LLM-Powered Lesson Summaries — a brand new worker, brand new event

This is the clearest demonstration of "new AI feature = new worker, no core changes" in the whole series [9]. Lesson summaries don't hook into `assignment.submitted` at all — they need a new event, `lesson.completed`, which we register exactly the same way we registered every worker since Part 6 [10]:

```json
{
  "name": "Summary Worker",
  "enabled": true,
  "events": ["lesson.completed"],
  "endpoint": "http://localhost:4003/api/summary-worker"
}
```
[10]

```typescript
// workers/summary-worker/index.ts
import express from "express";
import OpenAI from "openai";
import { verifySignature, signPayload } from "../../packages/workers/sign";

const app = express();
app.use(express.json());
const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

app.post("/api/summary-worker", async (req, res) => {
  const signature = req.headers["x-greymatter-signature"] as string;
  if (!verifySignature(req.body, signature)) return res.status(401).json({ error: "Invalid signature" });

  const completion = await openai.chat.completions.create({
    model: "gpt-4o-mini",
    messages: [
      { role: "system", content: "Summarize this lesson content in 3 concise bullet points for a student reviewing it." },
      { role: "user", content: req.body.lessonContent },
    ],
  });

  const output = {
    workerName: "summary-worker",
    resultType: "summary",
    data: { summary: completion.choices[0].message.content },
    success: true,
  };

  res.setHeader("x-greymatter-signature", signPayload(output, process.env.WORKER_SIGNING_SECRET!));
  res.json(output);
});

app.listen(4003, () => console.log("Summary Worker listening on :4003"));
```
[10]

To make this real end-to-end, we need one small addition: a place that actually emits `lesson.completed`. Add a Server Action alongside `submitAssignment` from Part 5 [5]:

```typescript
// src/app/(dashboard)/courses/actions.ts
"use server";

import { auth } from "@clerk/nextjs/server";
import { inngest } from "../../../../../infra/inngest/client";
import { newTraceId } from "../../../../../packages/events/trace";

export async function completeLesson(lessonId: string) {
  const { userId, orgId } = await auth();
  if (!userId || !orgId) throw new Error("Unauthorized");

  await inngest.send({
    name: "lesson.completed",
    data: { lessonId, userId, orgId, traceId: newTraceId() },
  });
}
```

You'll also need a small Inngest function to fetch the lesson content and pass it to the registry-discovered workers — following the exact same fetch → discover → execute → persist shape from Part 5 [5], just triggered by a different event.

**✅ Checkpoint:** Register the Summary Worker document in Sanity Studio (same steps as the Grading Worker checkpoint in Part 6 [4]), trigger `completeLesson` for a lesson with real content, and confirm a new `worker_results` row appears with `resultType: "summary"` containing three genuine bullet points — proving a feature with a brand-new event and a brand-new worker required zero changes to `packages/registry`, `infra/inngest/client.ts`, or the API route from Part 5.
