## 2. AI Quiz Generation Pipeline

Recall from Part 6 that we registered a "Quiz Worker" in Sanity against the `assignment.submitted` event, but it was never actually built — only referenced. Let's build the real thing now, replacing the placeholder from Part 5/6.

```bash
mkdir -p workers/quiz-worker
cd workers/quiz-worker
pnpm init
pnpm add express openai
```

```typescript
// workers/quiz-worker/index.ts
import express from "express";
import OpenAI from "openai";
import { verifySignature, signPayload } from "../../packages/workers/sign";
import type { WorkerInput, WorkerOutput } from "../../packages/workers/types";

const app = express();
app.use(express.json());
const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

app.post("/api/quiz-worker", async (req, res) => {
  const signature = req.headers["x-greymatter-signature"] as string;
  if (!verifySignature(req.body, signature)) {
    return res.status(401).json({ error: "Invalid signature" });
  }

  const input = req.body as WorkerInput;

  const completion = await openai.chat.completions.create({
    model: "gpt-4o-mini",
    messages: [
      { role: "system", content: "Generate 3 multiple-choice quiz questions based on this student submission content. Return JSON only." },
      { role: "user", content: input.submission.content },
    ],
  });

  const quiz = JSON.parse(completion.choices[0].message.content ?? "{}");

  const output: WorkerOutput = {
    workerName: "Quiz Worker",
    resultType: "quiz",
    data: { quiz },
    success: true,
    costCents: Math.round((completion.usage?.total_tokens ?? 0) * 0.001), // rough estimate
  };

  res.setHeader("X-Greymatter-Response-Signature", signPayload(output));
  res.json(output);
});

app.listen(4001, () => console.log("Quiz Worker listening on port 4001"));
```

**✅ Checkpoint:** Update the Quiz Worker's `endpoint` in Sanity Studio (Part 6) to `http://localhost:4001/api/quiz-worker`, then re-enable it if you disabled it earlier. Submit an assignment. In `worker_results` (Drizzle Studio), confirm a real row with `resultType: "quiz"` containing actual AI-generated questions, and a non-zero `costCents` value — this is the cost-tracking column we scaffolded in Part 10 finally being used for real.

## 3. Tutor AI (Adaptive Intervention System)

Recall from Part 8 that our `studentStruggling` function only logged a placeholder message: `"Tutor AI intervention triggered..."`. Let's replace that with real AI output — personalized explanations, learning path adjustments, and remediation suggestions [10].

```typescript
// workers/tutor-worker/index.ts
import express from "express";
import OpenAI from "openai";
import { verifySignature, signPayload } from "../../packages/workers/sign";

const app = express();
app.use(express.json());
const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

app.post("/api/tutor-worker", async (req, res) => {
  const signature = req.headers["x-greymatter-signature"] as string;
  if (!verifySignature(req.body, signature)) {
    return res.status(401).json({ error: "Invalid signature" });
  }

  const { studentId, submissionId, score } = req.body.data;

  const completion = await openai.chat.completions.create({
    model: "gpt-4o-mini",
    messages: [
      { role: "system", content: "You are a supportive AI tutor. A student scored low on an assignment. Provide a personalized explanation, a learning path adjustment, and a remediation suggestion." },
      { role: "user", content: `Student score: ${score}/100. Provide feedback.` },
    ],
  });

  const output = {
    workerName: "Tutor Worker",
    resultType: "tutor_feedback",
    data: {
      explanation: completion.choices[0].message.content,
      studentId,
      submissionId,
    },
    success: true,
  };

  res.setHeader("X-Greymatter-Response-Signature", signPayload(output));
  res.json(output);
});

app.listen(4002, () => console.log("Tutor Worker listening on port 4002"));
```

Now wire it into the `studentStruggling` Inngest function we built in Part 8, replacing the console.log placeholder:

```typescript
// infra/inngest/functions/studentStruggling.ts (updated)
import { inngest } from "../client";
import { signPayload } from "../../../packages/workers/sign";
import { db } from "../../db";
import { workerResults } from "../../db/schema";
import { internalEmit } from "../internalEmit";

export const studentStruggling = inngest.createFunction(
  { id: "student-struggling" },
  { event: "student.struggling" },
  async ({ event, step }) => {
    const tutorResult = await step.run("trigger-tutor-intervention", async () => {
      const payload = { event: "student.struggling", data: event.data };
      const signature = signPayload(payload);
      const response = await fetch("http://localhost:4002/api/tutor-worker", {
        method: "POST",
        headers: { "Content-Type": "application/json", "X-Greymatter-Signature": signature },
        body: JSON.stringify(payload),
      });
      return response.json();
    });

    await step.run("persist-tutor-feedback", async () => {
      await db.insert(workerResults).values({
        submissionId: event.data.submissionId,
        workerName: "Tutor Worker",
        resultType: "tutor_feedback",
        resultData: JSON.stringify(tutorResult.data),
      });
    });

    await internalEmit("practice.assigned", {
      studentId: event.data.studentId,
      submissionId: event.data.submissionId,
    });
  }
);
```

## 4. LLM-Powered Lesson Summaries

A new event, `lesson.completed`, gives us a natural place to add a Summary Worker. Register it in Sanity Studio (same pattern as Part 6):

```json
{
  "name": "Summary Worker",
  "enabled": true,
  "events": ["lesson.completed"],
  "endpoint": "http://localhost:4003/api/summary-worker"
}
```

```typescript
// workers/summary-worker/index.ts
import express from "express";
import OpenAI from "openai";
import { verifySignature, signPayload } from "../../packages/workers/sign";

const app = express();
app.use(express.json());
const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

app.post("/api/summary-worker", async (req, res) => {
  const signature = req.headers["x-greymatter-signature"] as string;
  if (!verifySignature(req.body, signature)) return res.status(401).json({ error: "Invalid signature" });

  const completion = await openai.chat.completions.create({
    model: "gpt-4o-mini",
    messages: [
      { role: "system", content: "Summarize this lesson content in 3 concise bullet points." },
      { role: "user", content: req.body.lessonContent },
    ],
  });

  const output = {
    workerName: "Summary Worker",
    resultType: "summary",
    data: { summary: completion.choices[0].message.content },
    success: true,
  };

  res.setHeader("X-Greymatter-Response-Signature", signPayload(output));
  res.json(output);
});

app.listen(4003, () => console.log("Summary Worker listening on port 4003"));
```

Add the matching Inngest function and emit event, following the exact same pattern established in Part 5 for `assignment.submitted` — a new event, a new function, no core changes required to any existing code, exactly as promised back in Part 5: new AI feature = new worker, no core changes [5].

// Inside summary-worker, an additional call
const graphCompletion = await openai.chat.completions.create({

We won't build a visualization UI for this in the core series (that's a great capstone extension exercise), but the data pipeline is now identical to every other worker we've built.

## 6. Why none of this required touching the LMS core

Stop and notice something important: across this entire part, we did not modify `assignmentSubmitted.ts`'s core logic, the Registry client from Part 6, the Worker SDK contract from Part 7, or any Server Action from Part 3. We only:

1. Built new workers (Quiz, Tutor, Summary)
2. Registered them in Sanity
3. Let Inngest's existing discovery + fan-out mechanism find and call them

This is the entire architectural payoff promised since Part 0 — AI becomes modular, each AI system is replaceable [4], and the LMS logic stays fully decoupled from intelligence logic [4].
