# Part 11 — AI-Native Features: Auto Summaries, Quiz Generation & Learning Intelligence 

In Part 10, we built tracing, distributed logging, and the groundwork for per-worker cost tracking — closing the "black box AI behavior" problem for good [11]. Every worker we've called since Part 5 has been faking its intelligence: a `Math.random()` score, a hardcoded quiz array, a canned tutor message. This part is the payoff of everything since Part 0 — we replace every one of those placeholders with real AI [10].

**🎯 Goal of this lesson:** Build LLM-powered lesson summaries, an automatic quiz generation system, adaptive learning insights, student weakness detection, an AI tutoring layer, and a light preview of knowledge graph extraction from LMS data — all using the exact same registration flow, signing scheme, and observability pipeline we've already built [10].

---

## 1. Why this part changes almost nothing structurally

This is worth saying plainly before writing any code: nothing about the six-step registration flow from Part 7 changes, nothing about HMAC signing changes, and nothing about the `WorkerInput`/`WorkerOutput` contract changes. The entire point of separating the Execution Layer from everything else since Part 1 was so that swapping a placeholder for real intelligence would be a change *inside* a worker's handler function, not a change to the system around it. If this part required touching `infra/inngest`, `packages/registry`, or the signing logic in `packages/workers`, that would mean the architecture had failed its own design goal.

## 2. Real grading — replacing the Grading Worker's placeholder score

Recall the Grading Worker built in Part 7 returned a hardcoded score and canned feedback string as a deliberate placeholder. Swap only the inside of its handler:

```typescript
// workers/grading-worker/index.ts (updated handler)
import OpenAI from "openai";

const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

app.post("/api/grading-worker", async (req, res) => {
  const signature = req.header("x-signature") ?? "";
  const input: WorkerInput = req.body;

  if (!verifySignature(input, signature)) {
    return res.status(401).json({ error: "Invalid signature" });
  }

  const completion = await openai.chat.completions.create({
    model: "gpt-4o-mini",
    messages: [
      { role: "system", content: "Grade this student submission from 0-100 and give one sentence of feedback. Respond as JSON: { score: number, feedback: string }" },
      { role: "user", content: input.content },
    ],
  });

  const { score, feedback } = JSON.parse(completion.choices[0].message.content ?? "{}");

  const output: WorkerOutput = {
    workerName: "grading-worker",
    resultType: "grading",
    data: { score, feedback },
    success: true,
  };

  res.setHeader("x-signature", signPayload(output, process.env.WORKER_SIGNING_SECRET!));
  res.json(output);
});
```

Nothing about the surrounding request — the signature check, the response shape, the endpoint registered in Sanity — changed at all.

**✅ Checkpoint:** Submit a genuinely thoughtful assignment and a deliberately low-effort one, and confirm the two get meaningfully different scores and feedback text — no longer the fixed `85` from earlier parts.

## 4. Adaptive learning insights and student weakness detection

With real grading feeding real scores into the conditional branch built in Part 8 — the check that emits `student.struggling` when `report.score < 70` [2] — we can now make that signal genuinely meaningful rather than a placeholder threshold. A weakness-detection pass reads a student's recent `worker_results` rows and surfaces recurring low-scoring topics, giving the `student.struggling` event a real, data-backed reason to fire rather than a single arbitrary score check.

## 5. AI tutoring — replacing the canned intervention message

Recall Part 8's `student-struggling` function returned a hardcoded intervention message as a placeholder [2]. Replace it with a real, context-aware LLM call:

```typescript
// infra/inngest/functions/studentStruggling.ts (tutor-intervention step, updated)
const intervention = await step.run("tutor-intervention", async () => {
  const completion = await openai.chat.completions.create({
    model: "gpt-4o-mini",
    messages: [
      { role: "system", content: "You are a supportive tutor. Write two encouraging sentences addressing this student's low score, and one concrete suggestion." },
      { role: "user", content: `Score: ${event.data.score}` },
    ],
  });
  return { message: completion.choices[0].message.content };
});
```

Note this is a change *inside* an existing Inngest function, not a new worker — tutor intervention lives in the Orchestration Layer's own logic in this design, rather than as a separate registered worker, which is a deliberate simplification worth noting as you build [10].

**✅ Checkpoint:** Force a low score again (as in Part 8's conditional-branch checkpoint [2]) and confirm the `tutor-intervention` step now returns a genuinely written, context-aware message — then check its `traceId` in the Part 10 timeline view to confirm it's still linked to the original submission [11].

## 6. Knowledge Graph Extraction — a light preview

The final and most advanced feature in this part is knowledge graph extraction from LMS data [10]. At this stage, treat it as a variant of the Summary Worker that additionally extracts structured concept relationships:

```typescript
// Inside summary-worker, an additional call
const graphCompletion = await openai.chat.completions.create({
  model: "gpt-4o-mini",
  messages: [
    { role: "system", content: "Extract key concepts from this lesson and their relationships as JSON: { nodes: [...], edges: [...] }" },
    { role: "user", content: req.body.lessonContent },
  ],
});
```

This output can be persisted the same way as everything else in Greymatter LMS — a new row in `worker_results` with `resultType: "knowledge_graph"` [10]. We won't build a visualization UI for this in the core series; it's flagged here explicitly as a capstone extension exercise, revisited again in Part 12 [9].
