# GreyMatter LMS — Data Dictionary

**Document type:** Data Dictionary
**Product:** GreyMatter LMS
**Version:** 1.0 (reflects implemented system, Parts 0–16)
**Status:** Baseline — approved
**Location:** `docs/DATA_DICTIONARY.md`
**Companion documents:** `docs/SRD.md`, `docs/ARCHITECTURE.md`, Appendix B, Appendix C

---

## 1. Purpose and Scope

This data dictionary is the authoritative, field-by-field reference for every data element in GreyMatter LMS — across both storage systems (Neon PostgreSQL and Sanity) and every derived/transient data shape that flows between them (TypeScript interfaces, Zod-validated inputs, Inngest event payloads). Where `docs/SRD.md` specifies *behavior* and Appendix B/C specify *schema structure*, this document specifies the precise **meaning, domain, format, and lifecycle of every individual field** — the level of detail needed to answer "what exactly can this field contain, where does its value come from, and what happens to it over time?"

This document is organized into six parts:

- **Section 2** — Neon PostgreSQL entities (the transactional system of record)
- **Section 3** — Sanity content entities (the content system)
- **Section 4** — Cross-system identifier fields (the deliberate seam between the two systems)
- **Section 5** — Derived/transient data shapes (validation inputs, event payloads, computed results)
- **Section 6** — Controlled vocabularies (every enum and closed string-union, consolidated)

## 2. Neon PostgreSQL Entities

### 2.1 Entity: `users`

**Description:** The internal identity record for every account on the platform, bridging the external identity provider to every other piece of transactional state.

| Field | Data type | Domain / format | Nullable | Default | Source of value | Lifecycle notes |
|---|---|---|---|---|---|---|
| `id` | UUID | RFC 4122 v4 UUID | No | Auto-generated by database | Generated at insert time | Immutable; the primary key referenced by every other table's `user_id` foreign key |
| `auth_provider_id` | Text | Clerk user ID format, e.g. `user_2abc...` | No | — | Supplied by Clerk (webhook payload or direct API fetch) | Immutable once set; unique across all rows |
| `email` | Text | RFC 5322-conformant email address | No | — | Supplied by Clerk (primary email address at time of sync) | Mutable — updated on `user.updated` webhook events; unique across all rows |
| `role` | Enum (`user_role`) | `STUDENT` \| `INSTRUCTOR` \| `ADMIN` | No | `STUDENT` | Set at creation; changed only via administrative action | See Section 6.1 for full enum definition |
| `created_at` | Timestamp with timezone | ISO 8601 | No | Current time at insert | Set once, at row creation | Immutable |
| `updated_at` | Timestamp with timezone | ISO 8601 | No | Current time at insert | Refreshed on any update | Mutable |

**Business rules:** `auth_provider_id` and `email` are each independently unique across the entire table. Deletion cascades to `enrollments`, `lesson_progress`, `course_progress`, `module_attempts`, `certificates`, `notifications`, `notification_preferences`; deletion sets `audit_logs.user_id` to null rather than cascading.

### 2.2 Entity: `enrollments`

**Description:** Records that a specific user has active (or formerly active) access to a specific course's authenticated learning content.

| Field | Data type | Domain / format | Nullable | Default | Source of value | Lifecycle notes |
|---|---|---|---|---|---|---|
| `id` | UUID | RFC 4122 v4 UUID | No | Auto-generated | Generated at insert | Immutable |
| `user_id` | UUID | Must match an existing `users.id` | No | — | The authenticated requester at time of enrollment | Immutable |
| `course_id` | Text | A Sanity course document `_id` (not database-enforced) | No | — | Supplied by client, independently re-verified against Sanity before insert | Immutable |
| `status` | Enum (`enrollment_status`) | `ACTIVE` \| `COMPLETED` \| `CANCELLED` | No | `ACTIVE` | Set at creation; transitioned to `COMPLETED` automatically upon certificate issuance | See Section 6.2 |
| `enrolled_at` | Timestamp with timezone | ISO 8601 | No | Current time at insert | Set once | Immutable |

**Business rules:** The combination `(user_id, course_id)` is unique — enforced at the database level, not solely by application logic. `course_id` has no database-level referential integrity with Sanity; every read/write path must independently verify the referenced course's existence and publication state.

### 2.3 Entity: `lesson_progress`

**Description:** Fine-grained, per-lesson completion tracking for a specific user within a specific enrollment.

| Field | Data type | Domain / format | Nullable | Default | Source of value | Lifecycle notes |
|---|---|---|---|---|---|---|
| `id` | UUID | RFC 4122 v4 UUID | No | Auto-generated | Generated at insert | Immutable |
| `user_id` | UUID | Must match `users.id` | No | — | The authenticated user | Immutable |

| `course_id` | Text | Sanity course `_id` | No | — | Denormalized copy of the enrollment's `course_id` | Immutable |
| `lesson_id` | Text | Sanity lesson `_id` | No | — | Supplied by the module-submission or lesson-view path | Immutable |
| `status` | Enum (`lesson_status`) | `NOT_STARTED` \| `IN_PROGRESS` \| `COMPLETED` | No | `NOT_STARTED` | `IN_PROGRESS` set upon first module attempt; `COMPLETED` set once every module in the lesson has a recorded attempt | See Section 6.3 |
| `completed_at` | Timestamp with timezone | ISO 8601 | Yes | `null` | Set only when `status` transitions to `COMPLETED` | Mutable until completion, then immutable |
| `created_at` | Timestamp with timezone | ISO 8601 | No | Current time at insert | Set once | Immutable |
| `updated_at` | Timestamp with timezone | ISO 8601 | No | Current time at insert | Refreshed on any update | Mutable |

**Business rules:** The combination `(user_id, lesson_id)` is unique. Writes use an upsert pattern (`onConflictDoUpdate`) keyed on this exact pair.

| Field | Data type | Domain / format | Nullable | Default | Source of value | Lifecycle notes |
|---|---|---|---|---|---|---|

### 2.5 Entity: `module_attempts`

**Description:** The most granular record in the system — one row per graded (or acknowledged) submission to one interactive assessment module.

| Field | Data type | Domain / format | Nullable | Default | Source of value | Lifecycle notes |
|---|---|---|---|---|---|---|
| `id` | UUID | RFC 4122 v4 UUID | No | Auto-generated | Generated at insert | Immutable |
| `user_id` | UUID | Must match `users.id` | No | — | The authenticated user | Immutable |
| `lesson_id` | Text | Sanity lesson `_id` | No | — | Supplied with the submission, verified against the scoped assessment lookup | Immutable |
| `module_id` | Text | Matches an authored `moduleId` field in Sanity | No | — | Supplied with the submission | Immutable |
| `attempt_number` | Integer | Positive integer, starting at `1` | No | `1` | Computed as `(prior attempt count for this user+module) + 1` at submission time | Immutable once written |
| `submission` | JSONB | Shape varies by module type — see Section 5.2 | No | — | The learner's raw input, as submitted by the client | Immutable |
| `score` | Integer | `0`–`100` inclusive | Yes | `null` | Computed exclusively by the server-side grading function | Immutable once written; `null` signifies "not applicable to this module type," distinct from a genuine zero |
| `is_correct` | Boolean | `true` \| `false` \| `null` | Yes | `null` | Computed exclusively by the server-side grading function | Immutable once written; `null` signifies "not applicable" |
| `idempotency_key` | Text | Client-generated UUID | Yes | `null` | Generated client-side, once per logical submission attempt | Immutable |
| `submitted_at` | Timestamp with timezone | ISO 8601 | No | Current time at insert | Set once | Immutable |

**Business rules:** The combination `(user_id, module_id, attempt_number)` is unique. The combination `(user_id, module_id, idempotency_key)` is unique when `idempotency_key` is non-null (Postgres treats multiple `NULL`s as non-conflicting). `score`/`is_correct` must never be populated from any client-supplied value under any circumstance — see `docs/ARCHITECTURE.md` §7.2 for the full security rationale.

### 2.6 Entity: `certificates`

**Description:** Immutable, historically-accurate proof of course completion.

| Field | Data type | Domain / format | Nullable | Default | Source of value | Lifecycle notes |
|---|---|---|---|---|---|---|
| `id` | UUID | RFC 4122 v4 UUID | No | Auto-generated | Generated at insert | Immutable |
| `user_id` | UUID | Must match `users.id` | No | — | The completing user | Immutable |
| `course_id` | Text | Sanity course `_id` | No | — | Copied from the completed enrollment | Immutable |
| `certificate_number` | Text | Format: `GM-YYYY-NNNNNN` (year + zero-padded 6-digit sequence value) | No | — | Generated from the atomic `certificate_number_seq` database sequence | Immutable; globally unique |
| `course_title` | Text | Free text, up to the course's authored title length | No | — | **Snapshot** — the course's title as read from Sanity at the moment of issuance | Immutable — never updated even if the live course is later renamed |
| `recipient_email` | Text | RFC 5322-conformant email address | No | — | **Snapshot** — the user's email as read from Neon at the moment of issuance | Immutable — never updated even if the account's email later changes |
| `issued_at` | Timestamp with timezone | ISO 8601 | No | Current time at insert | Set once | Immutable |

**Business rules:** The combination `(user_id, course_id)` is unique — enforced at the database level; issuance logic additionally performs an application-level check-then-insert-with-race-recovery pattern. `certificate_number` is never reused, even if a certificate row were ever deleted (the underlying sequence never resets or reuses a value).

### 2.7 Entity: `webhook_events`

**Description:** An idempotency ledger for every externally-delivered webhook received by the system.

| Field | Data type | Domain / format | Nullable | Default | Source of value | Lifecycle notes |
|---|---|---|---|---|---|---|
| `id` | UUID | RFC 4122 v4 UUID | No | Auto-generated | Generated at insert | Immutable |
| `source` | Text | Currently only `"clerk"`; extensible for future providers | No | — | Hardcoded per webhook route | Immutable |
| `event_type` | Text | Provider-defined event name, e.g. `"user.created"` | No | — | Read from the verified webhook payload | Immutable |
| `external_id` | Text | The provider's own unique delivery identifier (e.g., Svix's `svix-id`) | No | — | Read from the webhook's delivery headers | Immutable |
| `payload` | JSONB | The full, verified webhook body | No | — | Read from the verified request body | Immutable |
| `processed_at` | Timestamp with timezone | ISO 8601 | Yes | `null` | Set once associated processing completes successfully | Mutable — transitions from `null` to a real timestamp exactly once |
| `created_at` | Timestamp with timezone | ISO 8601 | No | Current time at insert | Set once | Immutable |

**Business rules:** The combination `(source, external_id)` is unique — this is the exact mechanism preventing double-processing of a redelivered webhook. The row is inserted **before** any associated business logic executes, to close the race window between "check if seen" and "record as seen."

## 4. Cross-System Identifier Fields (The Content/Transactional Seam)

These fields are the deliberate junction points between the two storage systems. Each is documented once here because their behavior is architecturally significant and repeats across multiple Neon tables.

| Field name (appears in) | Points to | Enforced by | Verification responsibility |
|---|---|---|---|
| `enrollments.course_id` | `course._id` | Nothing at the database level | Application re-verifies existence + `isPublished` at every enrollment write |
| `lesson_progress.course_id`, `lesson_progress.lesson_id` | `course._id`, `lesson._id` | Nothing at the database level | Application uses the course-scoped lesson query pattern to prove the relationship |
| `course_progress.course_id`, `course_progress.last_visited_lesson_id` | Same as above | Nothing at the database level | Same as above |
| `module_attempts.lesson_id`, `module_attempts.module_id` | `lesson._id`, an authored `moduleId` value | Nothing at the database level | Application uses the server-only assessment definition query, scoped through course → chapters → lessons → content |
| `certificates.course_id` | `course._id` | Nothing at the database level | Verified only at issuance time, then snapshotted — no longer re-verified afterward |
| `instructor.userId` | `users.id` | Nothing at the database level (Sanity has no knowledge of Neon) | Verified at every instructor-scoped resource request via `verifyCourseOwnership` |

**The universal rule governing every field in this section:** a value here is a plain string with no referential-integrity guarantee whatsoever. Every single read or write path touching one of these fields must independently prove the relationship it implies, using the scoped-query pattern documented in `docs/ARCHITECTURE.md` §7.3, before treating the reference as valid.

## 5. Derived / Transient Data Shapes

These shapes exist only in application memory or on the network — never persisted verbatim — but are precisely specified because they cross trust boundaries.

### 5.1 `CurrentUser` (resolved identity object)

| Field | Type | Description |
|---|---|---|
| `id` | String (UUID) | The internal Neon `users.id` |
| `authProviderId` | String | The Clerk identifier |
| `email` | String | The synced email |
| `role` | `UserRole` | One of `STUDENT` \| `INSTRUCTOR` \| `ADMIN` |

### 5.2 Module Submission Shapes (client → server, per module type)

| Module type | Submission shape | Notes |
|---|---|---|
| Quiz | `{ selectedOptionIndex: number }` | The zero-based index of the chosen option |
| Code exercise | `{ responseText: string }` | Max 5,000 characters (serialized JSON size limit) |
| Reflection | `{ responseText: string }` | Same size limit; no correctness implied |
| Checkpoint | `{ acknowledged: true }` | A fixed literal — no meaningful variation possible |

### 5.3 `ModuleSubmissionResult` (server → client)

| Field | Type | Description |
|---|---|---|
| `success` | Boolean | Whether the submission was accepted and processed |
| `isCorrect` | `boolean \| null` | `null` for module types with no correctness concept |

| `message` | String | Human-readable, safe (never raw error detail) |

## 6. Controlled Vocabularies (Enums and Closed Unions)

### 6.1 `user_role`

| Value | Meaning |
|---|---|
| `STUDENT` | Default role for every new account; may enroll and learn |
| `INSTRUCTOR` | May access analytics/roster tools for owned courses |
| `ADMIN` | Reserved for platform governance; route protection implemented, dedicated UI deferred |

### 6.2 `enrollment_status`

| Value | Meaning |
|---|---|
| `ACTIVE` | Enrollment in force; course content accessible |
| `COMPLETED` | Set automatically upon certificate issuance |
| `CANCELLED` | Reserved for a future refund/withdrawal workflow; enum value exists and is respected by every enrollment-status filter, though no current code path sets it |

### 6.3 `lesson_status`

| Value | Meaning |
|---|---|
| `NOT_STARTED` | Default; no attempt recorded yet |
| `IN_PROGRESS` | At least one module attempt recorded, not all modules attempted |
| `COMPLETED` | Every interactive module within the lesson has at least one recorded attempt |

### 6.4 `workflow_status`

| Value | Meaning |
|---|---|
| `PENDING` | Recorded at workflow start, before processing begins |
| `PROCESSED` | Workflow completed successfully |
| `FAILED` | Workflow terminated with an error; recorded before re-raising for engine-level retry |

### 6.5 `notification_type`

| Value | Meaning |
|---|---|
| `INACTIVITY_REMINDER` | Sent by the scheduled inactivity workflow, or manually by an instructor |
| `WEEKLY_DIGEST` | Sent by the scheduled digest workflow |
| `COURSE_COMPLETED` | Reserved for completion-related notification (issued alongside certificate email) |
| `NEW_CONTENT_PUBLISHED` | Reserved for future use — no current workflow emits this type |

### 6.6 `difficulty` (Sanity, course document)

| Value | Meaning |
|---|---|
| `beginner` | No prior knowledge assumed |
| `intermediate` | Some foundational knowledge assumed |
| `advanced` | Significant prior knowledge assumed |

### 6.7 `ModuleErrorCode` (application-level, not persisted)

| Value | Meaning |
|---|---|
| `NOT_ENROLLED` | The requester lacks an active enrollment for the claimed course |
| `MODULE_NOT_FOUND` | The claimed module/lesson/course chain could not be resolved |
| `ATTEMPT_LIMIT_EXCEEDED` | The configured maximum attempt count has been reached |
| `INVALID_SUBMISSION` | The submission failed shape validation for its module type |
| `SUBMISSION_TOO_LARGE` | The submission exceeded the configured size limit |
| `UNKNOWN_ERROR` | An unexpected failure occurred; no further detail is disclosed to the client |

## 7. Field Naming Conventions

| Convention | Example | Applies to |
|---|---|---|
| snake_case | `auth_provider_id`, `completion_percentage` | Neon column names (as stored) |
| camelCase | `authProviderId`, `completionPercentage` | TypeScript/Drizzle field references, Sanity field names |
| Suffix `_id` / `Id` | `course_id`, `courseId` | Any field holding a foreign key or cross-system identifier |
| Suffix `_at` / `At` | `enrolled_at`, `issuedAt` | Any timestamp field |
| Prefix `is` | `isPublished`, `isPreview`, `isCorrect` | Boolean flags |
| Suffix `Enabled` | `inactivityRemindersEnabled` | Boolean preference toggles specifically |

## 8. Change Control

Any modification to a field documented in this dictionary — a new column, a changed domain, an added enum value — requires:

1. A corresponding Drizzle schema change (Neon) or Studio schema change (Sanity), with a generated migration where applicable.
2. An update to this data dictionary in the same change set.
3. A review of `docs/SRD.md` Section 6 (Data Requirements) and Appendix F (Security Checklist) for any requirement implicated by the change — particularly for any field touching `module_attempts`, `certificates`, or any answer-key-bearing content field.
