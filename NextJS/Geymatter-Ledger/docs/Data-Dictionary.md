# 2. Global Data Conventions

## 2.1 Primary Keys

Most tables use:

```txt
id uuid primary key
```

Generated by Postgres:

```txt
gen_random_uuid()
```

---

## 2.2 Tenant Scope

Most business tables include:

```txt
organization_id
```

This is the primary tenant boundary.

A record with `organization_id` belongs to one company workspace.

---

## 2.3 Timestamps

Common timestamp fields:

```txt
created_at
updated_at
```

Timestamps use timezone-aware Postgres timestamps.

---

## 2.4 Money

Money is stored as integer cents.

Examples:

```txt
S$100.00 = 10000
S$9.00   = 900
```

Money columns usually end in:

```txt
_cents
```

---

## 2.5 Rates

Rates are stored as basis points.

Examples:

```txt
9% = 900
17% = 1700
20% = 2000
```

Rate columns usually end in:

```txt
_basis_points
```

---

## 2.6 Status Fields

Workflow states are represented with Postgres enums.

Examples:

```txt
invoice_status
bill_status
bank_transaction_status
```

# 9. Table: `journal_lines`

## Purpose

Stores debit/credit lines for journal entries.

This is the main report source.

---

## Columns

| Column | Type | Required | Description |
|---|---|---:|---|
| `id` | uuid | Yes | Journal line ID |
| `journal_entry_id` | uuid | Yes | Parent journal entry |
| `organization_id` | uuid | Yes | Owner organization |
| `account_id` | uuid | Yes | Account affected |
| `line_number` | integer | Yes | Order within entry |
| `description` | text | No | Line description |
| `debit_cents` | bigint | Yes | Debit amount in cents |
| `credit_cents` | bigint | Yes | Credit amount in cents |
| `created_at` | timestamptz | Yes | Creation time |

---

## Constraints

| Constraint | Meaning |
|---|---|
| debit non-negative | `debit_cents >= 0` |
| credit non-negative | `credit_cents >= 0` |
| exactly one side | Either debit or credit must be positive, not both |
| unique line number | `journal_entry_id + line_number` unique |

# 20. Cross-Table Integrity Checks

These SQL checks should return zero rows.

---

## Invoice Customer Organization Mismatch

```sql
select
  i.id,
  i.organization_id as invoice_org,
  c.organization_id as customer_org
from invoices i
join customers c
  on c.id = i.customer_id
where i.organization_id <> c.organization_id;
```

---

## Bill Vendor Organization Mismatch

```sql
select
  b.id,
  b.organization_id as bill_org,
  v.organization_id as vendor_org
from bills b
join vendors v
  on v.id = b.vendor_id
where b.organization_id <> v.organization_id;
```

---

## Journal Line Account Organization Mismatch

```sql
select
  jl.id,
  jl.organization_id as line_org,
  a.organization_id as account_org
from journal_lines jl
join accounts a
  on a.id = jl.account_id
where jl.organization_id <> a.organization_id;
```

---

# 21. Journal Balance Check

This SQL should return zero rows:

```sql
select
  je.id,
  je.memo,
  sum(jl.debit_cents) as debits,
  sum(jl.credit_cents) as credits,
  sum(jl.debit_cents) - sum(jl.credit_cents) as difference
from journal_entries je
join journal_lines jl
  on jl.journal_entry_id = je.id
group by je.id, je.memo
having sum(jl.debit_cents) <> sum(jl.credit_cents);
```

# 22. Final Notes

The most important data dictionary rules are:

```txt
organization_id protects tenant boundaries.
_cents fields store integer money.
_basis_points fields store rates.
journal_lines are the report source of truth.
source documents link to journal entries.
audit logs record important operations.
```

If a future table stores business or accounting data, it likely needs:

```txt
id
organization_id
created_at
updated_at
foreign keys
indexes
constraints
audit behavior
```
