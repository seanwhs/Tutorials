## Step 4 — Configuring Function Limits with `vercel.json`

### The Target
A new `vercel.json` file at the project root, explicitly configuring the memory and execution-time limits for our conversion Route Handler — since generating a PDF/DOCX/PPTX (especially one with several remote images to fetch, per Parts 5–7) can reasonably take longer than a typical API request.

### The Concept

> **Analogy — Booking Enough Time and a Big Enough Truck for a Large Delivery.** Every serverless function (the technical name for the isolated, on-demand unit of server code that runs each of our Route Handler requests — Appendix A covers this model in full) on Vercel has default limits on how long it's allowed to run and how much memory it can use, before being forcibly stopped. Vercel's Hobby (free) tier's defaults are tuned for typical, fast API responses — but our conversion endpoint does real, sometimes-slower work: parsing, tree-walking, font loading (Part 5D), and fetching remote images (Parts 5C/6C/7C). We explicitly configure a higher — but still free-tier-compatible — duration and memory allowance specifically for this one route, rather than leaving it at defaults tuned for simpler endpoints.

> **A note on exact numbers:** Vercel's specific free-tier limits (currently up to 60 seconds max duration and up to ~3GB memory configurable on Hobby, at time of writing) are a moving target and can change — always check your Vercel dashboard's current plan limits before assuming a specific number. We use conservative, safely-within-free-tier values below.

### The Implementation

**`vercel.json`**

```json
{
  "functions": {
    "app/api/convert/[format]/route.ts": {
      "maxDuration": 30,
      "memory": 1024
    }
  }
}
```

A few details worth pausing on:

- **`maxDuration: 30`** (seconds) gives our conversion endpoint significantly more time than a typical API's default (~10 seconds), accommodating slower cases like a document with several remote images needing to be fetched (Parts 5C/6C/7C's `fetchImages` functions), while staying comfortably within Hobby-tier's allowed maximum.
- **`memory: 1024`** (megabytes) is Vercel's own default value for most functions already, made explicit here intentionally — if you find real-world documents with many large embedded images cause memory-related failures once deployed, this is the exact value you'd increase first, and having it explicitly configured here (rather than left implicit) makes that future adjustment a one-line change.
- Only our conversion route is configured here — every other route in our app (`/`, `/inspector`) uses Vercel's ordinary defaults, since they don't perform any heavy binary generation work.

## Step 7 — Understanding the Real-World Failure Mode

### The Target
No code yet — building an accurate mental model of *what actually breaks* under stress, so the rate limiter and size guards we already built don't feel like arbitrary defensive code, but rather targeted fixes for specific, understood risks.

### The Concept

> **Analogy — A Single Toll Booth on a Highway.** Every serverless function invocation (recall Part 9A's definition: an isolated, on-demand unit of server code) is like one toll booth lane opening up briefly to process one car, then closing. If 500 cars (requests) arrive at once, Vercel opens many toll booths in parallel — but each individual booth (each single request handling one conversion) is still bounded by our `vercel.json` limits from 9A: 30 seconds max, 1024MB max memory. A single *enormous* Markdown document — say, one with 200 embedded remote images, right at our `MAX_MARKDOWN_LENGTH` ceiling from Part 8A — could plausibly threaten both limits simultaneously: fetching 200 images sequentially could exceed 30 seconds, and holding 200 images' worth of bytes in memory simultaneously (recall our `fetchImages` functions from Parts 5C/6C/7C, which use `Promise.all` to fetch *concurrently*, holding all results in memory at once) could threaten the 1024MB ceiling.

This is worth stating plainly: **our existing `MAX_MARKDOWN_LENGTH` guard from Part 8A bounds text size, but does not directly bound the *number of images* a document can reference** — a document could be relatively short in characters (well under 200,000) while still containing dozens of image URLs. This is a genuine, previously-unaddressed gap, and the right place to close it is exactly here, now that we understand the real production stakes.

## Step 8 — Capping the Number of Images Per Document

### The Target
A shared constant, `MAX_IMAGES_PER_DOCUMENT`, enforced identically across all three converters' `fetchImages` functions — the same "shared constant, enforced consistently" pattern from Part 8A's `MAX_MARKDOWN_LENGTH`.

### The Implementation

**`lib/constants.ts`** — add this new constant alongside `MAX_MARKDOWN_LENGTH`:

```typescript
export const MAX_MARKDOWN_LENGTH = 200_000;

/**
 * The maximum number of DISTINCT image URLs a single document is allowed
 * to reference. Bounds worst-case memory (all fetched images are held in
 * memory simultaneously via Promise.all — see Parts 5C/6C/7C) and
 * worst-case time (fetching many remote URLs, even concurrently, still
 * has a practical ceiling before it risks our 30-second vercel.json limit
 * from Part 9A) for a single conversion request.
 */
export const MAX_IMAGES_PER_DOCUMENT = 20;
```

Now update each converter's `fetchImages` function to enforce this cap. The change is identical in all three files — shown here for `lib/converters/toPdf.tsx` (apply the same change to `toDocx.ts` and `toPptx.ts`):

```typescript
import { MAX_IMAGES_PER_DOCUMENT } from "@/lib/constants";

async function fetchImages(ast: Root): Promise<Map<string, Buffer | null>> {
  const urls = new Set<string>();
  collectImageUrls(ast, urls);

  // Enforce the cap BEFORE fetching anything — if a document references
  // too many images, we treat every image beyond the limit as if it had
  // failed to load (our existing graceful-fallback rendering path from
  // Parts 5C/6C/7C already handles a `null` byte value correctly, so this
  // reuses that exact mechanism rather than requiring new fallback logic).
  const urlsArray = Array.from(urls);
  const urlsToFetch = urlsArray.slice(0, MAX_IMAGES_PER_DOCUMENT);
  const urlsSkipped = urlsArray.slice(MAX_IMAGES_PER_DOCUMENT);

  if (urlsSkipped.length > 0) {
    console.warn(
      `[toPdf] Document references ${urlsArray.length} images, exceeding the limit of ${MAX_IMAGES_PER_DOCUMENT}. ${urlsSkipped.length} image(s) will render as fallback placeholders.`
    );
  }

  const results = new Map<string, Buffer | null>();

  // Images beyond the cap are marked null immediately, with no network
  // request made for them at all — this is the actual resource savings,
  // not just a cosmetic warning.
  for (const url of urlsSkipped) {
    results.set(url, null);
  }

  await Promise.all(
    urlsToFetch.map(async (url) => {
      try {
        const response = await fetch(url);
        if (!response.ok) {
          console.warn(`[toPdf] Image fetch failed (status ${response.status}): ${url}`);
          results.set(url, null);
          return;
        }
        const arrayBuffer = await response.arrayBuffer();
        results.set(url, Buffer.from(arrayBuffer));
      } catch (err) {
        console.warn(`[toPdf] Image fetch threw an error for ${url}:`, err);
        results.set(url, null);
      }
    })
  );

  return results;
}
```

Repeat this identical pattern in `lib/converters/toDocx.ts` and `lib/converters/toPptx.ts` (adjusting only the `console.warn` prefix tag, e.g. `[toDocx]`/`[toPptx]`, to match each file's existing convention).

## Step 9 — Adding Basic Rate Limiting

### The Target
A simple, in-memory rate limiter protecting `/api/convert/[format]` from being hammered by a single client — appropriate for a small, free-tier-hosted public tool like ours.

### The Concept

> **Analogy — A Coffee Shop Limiting "One Espresso Shot Order Per Minute Per Customer."** Without any limit, one person (or one misbehaving script) could submit hundreds of conversion requests per second, each one spinning up real server work — parsing, font-fetching, image-fetching — potentially exhausting your free-tier's monthly usage allowance or degrading performance for genuine visitors. A rate limiter is a bouncer who tracks "how many requests has this specific visitor made recently," and politely declines ("come back in a moment") once they exceed a reasonable threshold — without needing to know anything about *why* the requests are happening.

We'll build a minimal **in-memory** limiter — tracking request counts in a plain JavaScript `Map`, kept entirely in server memory. This is explicitly a pragmatic choice appropriate for our app's actual scale (a small personal/portfolio tool), not a large-scale production system: it's simple, requires zero additional services or cost, and is honestly limited in one specific way we'll call out directly — worth understanding rather than hiding.

> **An honest limitation, stated upfront:** Vercel's serverless functions may run across multiple, separate underlying instances — an in-memory `Map` living inside one instance's memory doesn't share its counts with a *different* instance handling a different request. For genuinely high-traffic production rate limiting, a shared external store (like Vercel KV, Redis, or Upstash) would be the correct, robust solution. For GreyMatter MConvert's realistic scale as a personal or portfolio project, this simpler in-memory approach still meaningfully deters casual abuse and costs nothing extra — a deliberate, honest trade-off, not an oversight.

### The Implementation

**`lib/rateLimit.ts`**

```typescript
/**
 * A minimal, in-memory rate limiter: tracks how many requests a given
 * identifier (here, the requester's IP address) has made within a sliding
 * time window, and rejects requests beyond a configured threshold.
 *
 * See the honest limitation noted in Part 9C: this Map's state is local
 * to a single server instance and does not synchronize across multiple
 * instances — appropriate for this app's real-world scale, not a
 * guarantee against a determined, distributed attacker.
 */

interface RateLimitEntry {
  count: number;
  windowStart: number;
}

const requestCounts = new Map<string, RateLimitEntry>();

const WINDOW_MS = 60_000; // 1 minute sliding window
const MAX_REQUESTS_PER_WINDOW = 10; // 10 conversions per minute per IP

export interface RateLimitResult {
  allowed: boolean;
  retryAfterSeconds?: number;
}

export function checkRateLimit(identifier: string): RateLimitResult {
  const now = Date.now();
  const entry = requestCounts.get(identifier);

  if (!entry || now - entry.windowStart > WINDOW_MS) {
    // No prior entry, or the previous window has fully expired — start a
    // fresh window for this identifier.
    requestCounts.set(identifier, { count: 1, windowStart: now });
    return { allowed: true };
  }

  if (entry.count >= MAX_REQUESTS_PER_WINDOW) {
    const elapsedMs = now - entry.windowStart;
    const retryAfterSeconds = Math.ceil((WINDOW_MS - elapsedMs) / 1000);
    return { allowed: false, retryAfterSeconds };
  }

  entry.count += 1;
  return { allowed: true };
}

/**
 * Periodically clears stale entries so this Map doesn't grow unbounded
 * over a long-running server process — a small but genuine memory-hygiene
 * detail worth including explicitly rather than leaving as a slow leak.
 */
setInterval(() => {
  const now = Date.now();
  for (const [key, entry] of requestCounts.entries()) {
    if (now - entry.windowStart > WINDOW_MS) {
      requestCounts.delete(key);
    }
  }
}, WINDOW_MS).unref();
// .unref() tells Node.js this timer should NOT, by itself, keep the
// process alive — appropriate for a serverless environment where we never
// want a background timer to prevent a function instance from shutting
// down cleanly when it's otherwise idle.
```

Now wire this into the Route Handler. Update **`app/api/convert/[format]/route.ts`** — add the import and a new check, inserted immediately after the format validation (before any body parsing, since rejecting an over-limit request should happen as early and cheaply as possible):

```typescript
import { NextRequest, NextResponse } from "next/server";
import { renderToBuffer } from "@react-pdf/renderer";
import { parseMarkdown } from "@/lib/parseMarkdown";
import { toPdf } from "@/lib/converters/toPdf";
import { toDocx } from "@/lib/converters/toDocx";
import { sectionizeAst } from "@/lib/converters/sectionizeMarkdown";
import { toPptx } from "@/lib/converters/toPptx";
import { MAX_MARKDOWN_LENGTH } from "@/lib/constants";
import { checkRateLimit } from "@/lib/rateLimit";

export const runtime = "nodejs";

const SUPPORTED_FORMATS = ["pdf", "docx", "pptx"] as const;
type SupportedFormat = (typeof SUPPORTED_FORMATS)[number];

function isSupportedFormat(value: string): value is SupportedFormat {
  return (SUPPORTED_FORMATS as readonly string[]).includes(value);
}

const CONTENT_TYPES: Record<SupportedFormat, string> = {
  pdf: "application/pdf",
  docx: "application/vnd.openxmlformats-officedocument.wordprocessingml.document",
  pptx: "application/vnd.openxmlformats-officedocument.presentationml.presentation",
};

export async function POST(
  request: NextRequest,
  { params }: { params: Promise<{ format: string }> }
) {
  const { format } = await params;

  if (!isSupportedFormat(format)) {
    return NextResponse.json(
      {
        error: `Unsupported export format: "${format}". Supported formats are: ${SUPPORTED_FORMATS.join(", ")}.`,
      },
      { status: 400 }
    );
  }

  // Vercel (and most hosts) forward the real client IP via this header —
  // we fall back to a constant string in local development, where this
  // header typically isn't set, so rate limiting still "works" (treating
  // your local machine as one consistent identifier) without erroring.
  const identifier = request.headers.get("x-forwarded-for") ?? "local-dev";
  const rateLimitResult = checkRateLimit(identifier);

  if (!rateLimitResult.allowed) {
    return NextResponse.json(
      {
        error: `Too many requests. Please try again in ${rateLimitResult.retryAfterSeconds} seconds.`,
      },
      {
        status: 429, // "Too Many Requests" — the standard HTTP status for exactly this
        headers: { "Retry-After": String(rateLimitResult.retryAfterSeconds) },
      }
    );
  }

  let body: unknown;
  try {
    body = await request.json();
  } catch {
    return NextResponse.json(
      { error: "Request body must be valid JSON." },
      { status: 400 }
    );
  }

  if (
    typeof body !== "object" ||
    body === null ||
    !("markdown" in body) ||
    typeof (body as { markdown: unknown }).markdown !== "string"
  ) {
    return NextResponse.json(
      { error: "Request body must include a 'markdown' string field." },
      { status: 400 }
    );
  }

  const markdown = (body as { markdown: string }).markdown;

  if (markdown.trim().length === 0) {
    return NextResponse.json(
      { error: "Markdown content cannot be empty." },
      { status: 400 }
    );
  }

  if (markdown.length > MAX_MARKDOWN_LENGTH) {
    return NextResponse.json(
      {
        error: `Markdown content is too large (${markdown.length.toLocaleString()} characters). The maximum allowed size is ${MAX_MARKDOWN_LENGTH.toLocaleString()} characters.`,
      },
      { status: 413 }
    );
  }

  const ast = parseMarkdown(markdown);

  let fileBuffer: Buffer;

  try {
    if (format === "pdf") {
      const pdfElement = await toPdf(ast);
      fileBuffer = await renderToBuffer(pdfElement);
    } else if (format === "docx") {
      fileBuffer = await toDocx(ast);
    } else {
      const sections = sectionizeAst(ast);
      fileBuffer = await toPptx(sections);
    }
  } catch (err) {
    const message = err instanceof Error ? err.message : "Unknown rendering error.";
    console.error(`[convert/${format}] Rendering failed:`, err);
    return NextResponse.json(
      { error: `Failed to generate ${format.toUpperCase()} file: ${message}` },
      { status: 500 }
    );
  }

  const filename = `greymatter-export.${format}`;

  return new NextResponse(fileBuffer, {
    status: 200,
    headers: {
      "Content-Type": CONTENT_TYPES[format],
      "Content-Disposition": `attachment; filename="${filename}"`,
      "Content-Length": String(fileBuffer.byteLength),
    },
  });
}
```

A detail worth pausing on: **the rate limit check happens before JSON body parsing, and even before the size guard** — this ordering is deliberate. We want to reject an abusive burst of requests as cheaply as possible, before spending any effort parsing a request body that might itself be large or malformed. This mirrors the same "fail fast, fail cheap" instinct from Part 8A's size-guard ordering, just one layer earlier in the pipeline.
