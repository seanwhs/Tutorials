# Part 3 — Reference Appendix

Deep-dive material for Part 3, isolated from the step-by-step tutorial flow.

## A.1 Object storage concepts in more depth

Object storage systems like S3, MinIO, and Cloudflare R2 organize data into "buckets" (top-level containers, similar to a drive) holding "objects" (individual files), each addressed by a "key" (a string, often resembling a file path like `documents/sample.pdf`, though object storage has no real folder hierarchy underneath — the slashes are purely a naming convention most consoles display as folders for convenience).

Unlike a traditional filesystem, object storage has no concept of "opening a file, seeking to a byte offset, and writing in place" — objects are treated as atomic units: you PUT a whole object, or GET a whole object (or a specific byte range via HTTP Range headers, which some SDKs expose). This atomicity is actually convenient for our use case: Part 6's pattern of always writing a brand new object for any edit (rather than modifying an existing one) aligns naturally with how object storage is designed to be used.

## A.2 ReadableStream and Node.js streaming internals

A `ReadableStream` (the type returned by `getObjectStream` in Part 3) represents a source of data that arrives in chunks over time, rather than all at once. You consume it by calling `.getReader()` to obtain a reader object, then repeatedly calling `reader.read()`, which returns a Promise resolving to `{ done, value }` — `done` is true once the stream is exhausted, and `value` (when not done) is a chunk of data, typically a `Uint8Array` for binary streams like ours.

In `proxy.ts`, we hand this stream directly to `new NextResponse(stream, ...)` — Next.js's underlying HTTP server infrastructure consumes the stream on our behalf, writing each chunk to the actual network socket as it becomes available, rather than requiring us to manually loop over `reader.read()` ourselves for that specific use case. Part 6's `loadDocumentBytes`, by contrast, needs the complete bytes in memory (because pdf-lib cannot parse a PDF's structure incrementally), so it does perform that manual read loop explicitly, accumulating chunks into one final combined `Uint8Array`.

## A.3 A deeper security discussion: signed URLs vs. proxying

Part 3 chose "proxying" — every request for a document's bytes passes through our own Next.js server, which checks permissions and then streams the bytes itself. An alternative, commonly used pattern is "signed URLs" (also called pre-signed URLs): our server generates a temporary, cryptographically-signed URL pointing directly at the object in storage (e.g. directly at MinIO/S3), valid for a short window (e.g. 5 minutes), and hands that URL to the browser, which then fetches the file directly from storage, bypassing our Next.js server entirely for that download.

**Tradeoffs:** signed URLs reduce load on your application server (the file transfer happens directly between the browser and the storage provider, which is often better optimized for high-throughput file serving than a general-purpose app server), but they mean the storage provider's URL structure and bucket name become visible to the browser, and revoking access mid-download is impossible once a signed URL has been issued (it remains valid until its expiry, even if you change the document's permissions in your database a second later). Proxying (Part 3's choice) has the opposite tradeoffs: every byte flows through your app server (more load, but you already need application-server capacity for everything else Greymatter PDF does), but permission checks are enforced in real time on every single request, and no storage-provider details are ever exposed to the browser. For a security-sensitive document product where permissions might change frequently (e.g. revoking a shared document's access), proxying's real-time enforcement is generally the safer default — which is why this series chose it — though a mature production system might use signed URLs for less sensitive, rarely-changing assets (like public marketing PDFs) while keeping proxying for private user documents.

## A.4 Glossary of terms introduced in Part 3

- **Object storage**: a storage service organizing data into buckets and key-addressed objects, distinct from a filesystem or a database.
- **Bucket**: a top-level container for objects in an object storage system.
- **MinIO**: an open-source, S3-API-compatible object storage server, useful for local development without needing real cloud credentials.
- **ReadableStream**: a web-standard representation of data arriving in chunks over time.
- **Route Handler**: a Next.js App Router file (conventionally named `route.ts`) that responds to raw HTTP requests, as opposed to a Server Action called like a function from React code.
- **Signed URL**: a temporary, cryptographically-signed URL granting time-limited direct access to a storage object, bypassing the application server.
- **AcroForm** *(forward reference, formally introduced in Part 7)*: not used in Part 3, listed here only to avoid confusion with the unrelated "form" terminology if skimming ahead.

## A.6 Common pitfalls at this stage

1. **Forgetting `forcePathStyle: true`**: MinIO requires path-style addressing; omitting this option causes confusing connection errors when running against MinIO specifically (real AWS S3 tolerates either style, so this bug can hide until you test against MinIO).
2. **Returning different error messages for "not found" vs "forbidden"**: as explained in `proxy.ts`'s comments, this leaks which document IDs exist to anyone probing random IDs — always return the same generic message for both cases.
3. **Forgetting `Cache-Control: private, no-store`**: without this header, browsers or intermediate CDNs could cache a private document's response and serve it to a different user later.
4. **Missing environment variables**: since `s3-client.ts`'s `requireEnv` throws immediately at module load time if any `STORAGE_` variable is missing, a missing variable manifests as a crash the moment any file importing `s3-client.ts` is loaded, which can look alarming but is precisely the "fail loudly and immediately" behavior we want for a missing security-critical configuration value.
