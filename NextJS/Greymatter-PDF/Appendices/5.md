# Part 5 — Reference Appendix

Deep-dive material for Part 5, isolated from the step-by-step tutorial flow.

## A.1 Prisma migration workflows in more depth

`npx prisma migrate dev --name init` does two things: it generates a new SQL migration file inside `prisma/migrations/` (a timestamped folder containing the exact SQL statements needed to bring the database from its previous state to match your current `schema.prisma`), and it applies that SQL directly to your development database. Every subsequent schema change (adding a field, adding a new model) should go through the same command with a new descriptive name (e.g. `npx prisma migrate dev --name add-annotation-color`) — this produces a versioned history of every schema change, which is what lets a team (or a production deployment pipeline) apply the exact same sequence of changes to a different database (staging, production) via `npx prisma migrate deploy`, which applies pending migrations without prompting interactively (appropriate for automated deployment scripts, unlike `migrate dev` which is meant for local development).

A subtlety worth knowing: `migrate dev` will prompt to reset your database if it detects the migration history and the actual database state have diverged (e.g. you manually edited a table via `psql` outside of Prisma) — this is a safety mechanism, not a bug, ensuring your migration history remains a trustworthy, complete record of every change.

## A.2 JSON columns vs. normalized tables: a deeper tradeoff discussion

Part 5's schema stores an Annotation's `rect` and `points` fields as Postgres JSON columns rather than as separate, fully normalized tables (e.g. a separate `AnnotationRect` table with its own x/y/w/h columns, and a separate `AnnotationPoint` table with one row per point). The JSON approach is simpler to write and matches our TypeScript union type directly, but it has real costs: you cannot write a SQL WHERE clause efficiently filtering "all annotations whose rect.x is greater than 0.5" without Postgres-specific JSON operators (which exist, but are less ergonomic and less indexable than a plain column), and Prisma cannot type-check the internal shape of a Json field the way it types a normal column — our `rowToAnnotation` function's type casts (`row.rect as {...}`) are us manually asserting a shape Prisma itself does not verify at the database layer.

For Greymatter PDF's actual query patterns (always "get every annotation for this document," never "find annotations by their rect's specific x value"), the JSON approach is a reasonable, deliberate simplification. A system that needed to run analytical queries across annotation positions (e.g. "find all annotations in the top-right quadrant of any page, across all documents") would benefit from normalizing those fields into real columns instead, trading write-simplicity for query-power.

## A.3 The XFDF specification beyond what we implemented

Part 5 implemented `highlight`, `square` (mapped from our "rectangle" kind), and `ink` (mapped from "freehand") XFDF elements — a genuinely useful subset, but XFDF's full specification (published by Adobe and available in detail from the ISO 32000/PDF Association's documentation) additionally covers: text annotations (sticky-note-style comments not tied to a drawn shape), free text annotations (text boxes with their own visible border, distinct from filled AcroForm fields), line and polygon annotations, stamp annotations (predefined or custom image stamps, conceptually related to but formally distinct from the watermarking technique used in Part 7), and a `<fields>` section (separate from `<annots>`) representing AcroForm field values — meaning XFDF can actually represent Part 7's form-filling data too, not just drawn annotations, though this series' `xfdf.ts` module focuses specifically on the annotation-overlay use case from Part 4 rather than attempting full round-trip fidelity with every XFDF feature.

## A.4 Glossary of terms introduced in Part 5

- **ORM (Object-Relational Mapper)**: a library (Prisma, in our case) providing type-safe, function-call-based access to a relational database, instead of writing raw SQL strings by hand.
- **Migration**: a versioned, timestamped record of a specific database schema change, generated and applied by Prisma's migrate commands.
- **XFDF (XML Forms Data Format)**: an Adobe-published, industry-adopted XML standard for representing PDF annotations and form data outside of the PDF file itself.
- **Optimistic update**: a UI pattern where a change is reflected immediately in the interface before the corresponding server request completes, with a rollback mechanism if that request ultimately fails.
- **AcroForm**: *(forward reference, formally covered in Part 7)* a PDF's built-in interactive form field system.

## A.6 Common pitfalls at this stage

1. **Forgetting to run `migrate dev` after editing `schema.prisma`**: the Prisma Client's TypeScript types only regenerate after a migration runs (or after `npx prisma generate` alone, without a migration, if you only need updated types without a schema change) — editing the schema file alone does not update anything else.
2. **Committing the auto-generated `.env` file from `prisma init`**: Part 5 explicitly removes this file in favor of the established `.env.local` pattern; forgetting this step leaves two competing environment files that can cause confusing "which value actually applies" bugs.
3. **Not handling the optimistic-update rollback path**: if a Server Action call fails, forgetting to revert the optimistic local state change (as `use-annotations.ts`'s catch blocks do) leaves the UI showing data the server never actually accepted, silently diverging from reality.
4. **Assuming JSON column values are already correctly typed**: Prisma returns `Json`-typed columns as `unknown` or a very loose type in TypeScript; always validate/cast deliberately (as `rowToAnnotation` does) rather than assuming the stored shape matches your expectations without any runtime check.
