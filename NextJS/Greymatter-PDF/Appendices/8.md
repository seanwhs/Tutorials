# Part 8 — Reference Appendix

## A.1 LibreOffice's soffice command-line flags in more depth

Beyond `--headless` and `--convert-to pdf` used in Part 8, `soffice` supports several other flags worth knowing about for future extension: `--convert-to` accepts format-specific filter strings for finer control (e.g. `pdf:writer_pdf_Export` explicitly selects the Writer PDF export filter, useful when a file's extension alone is ambiguous about which LibreOffice component should handle it); `--outdir` specifies the output directory (used in Part 8 to direct output into our per-request temporary working directory); `-env:UserInstallation=file:///tmp/some-unique-path` can be used to give each concurrent conversion its own isolated LibreOffice user profile directory, which becomes important under concurrent load, since LibreOffice's headless mode has historically had issues with multiple simultaneous invocations sharing the same default profile directory and interfering with each other — a production deployment handling meaningful concurrent conversion volume should specifically research and test this concurrency behavior rather than assuming Part 8's single-request-at-a-time examples scale linearly without adjustment.

## A.2 Horizontal scaling strategies for the conversion service under real load

Because Part 8's conversion-service container is entirely stateless (every request is self-contained: receive file, convert, respond, clean up), it is a strong candidate for horizontal scaling — running multiple identical copies of the container behind a load balancer, with incoming conversion requests distributed across them. Container orchestration platforms (Kubernetes, AWS ECS, Google Cloud Run, and similar) generally support this pattern natively via a "replica count" or "minimum/maximum instances" configuration, often with automatic scaling based on CPU usage or request queue depth. The one added complexity beyond what a purely stateless service would need is LibreOffice's per-instance concurrency caveat from A.1 above — if a single container instance needs to handle more than one conversion simultaneously (rather than one request per container, with the orchestrator handling concurrency by running more container instances instead), you would need the `-env:UserInstallation` isolation technique to prevent simultaneous requests within the same container from interfering with each other.

## A.3 Security considerations for accepting arbitrary Office file uploads

Running LibreOffice against user-uploaded files carries a specific security consideration worth being explicit about: Office file formats (particularly older `.doc`/`.xls`/`.ppt` binary formats, and to a lesser extent modern XML-based `.docx`/`.xlsx`/`.pptx` formats) have historically been a vector for malicious files exploiting parser vulnerabilities in office suites, including LibreOffice itself. Part 8's Docker isolation (Section 8.1) is a meaningful defense-in-depth measure — a successful exploit is contained to the conversion container rather than compromising the main application server directly — but is not a complete solution on its own. A production deployment handling untrusted, public user uploads should additionally consider: running the conversion container with a non-root user and minimal filesystem write permissions, network-isolating the container (it needs no outbound internet access to do its job), keeping LibreOffice's package version current via regular image rebuilds (security patches for office-suite parsers are released periodically), and potentially scanning uploaded files with a dedicated malware-scanning tool before they ever reach the conversion service. These hardening measures are legitimate production concerns beyond this tutorial series' scope, but should not be skipped when actually deploying a system like this to handle files from untrusted members of the public.

## A.4 Glossary of terms introduced in Part 8

- **Headless**: running software without a graphical user interface, controlled entirely via command line or programmatic API.
- **Microservice**: a small, independently deployable service focused on one specific responsibility, communicating with other parts of a system over a network protocol (HTTP, in our case).
- **soffice**: LibreOffice's command-line executable, usable for headless document conversion among other tasks.
- **multer**: a widely-used Express.js middleware for parsing multipart/form-data file uploads.
- **Stateless service**: a service that retains no data between individual requests, making it straightforward to scale horizontally by running multiple identical instances.

## A.6 Common pitfalls at this stage

1. **Forgetting `--outdir`**: without explicitly specifying an output directory, `soffice` writes converted output alongside the input file, which can produce unexpected file locations if not explicitly controlled, as Part 8's `server.js` does.
2. **Not setting a timeout on the exec call**: a malformed or unusual input file can occasionally cause LibreOffice to hang rather than error out cleanly — Part 8's 60-second timeout on `execAsync` is a deliberate, necessary safeguard, not an arbitrary number.
3. **Assuming the container build is fast**: installing LibreOffice's packages downloads a genuinely large amount of data on first build — subsequent builds are faster due to Docker's layer caching, but the very first `docker compose up -d --build` can take several minutes, which can look like a hang if you are not expecting it.
4. **Testing only with well-formed files**: as Part 8's own verification Step 8 emphasizes, deliberately testing with a corrupted/malformed file is important — a conversion pipeline that only ever sees clean test files during development can hide error-handling gaps that only surface once real, messy user uploads arrive in production.
