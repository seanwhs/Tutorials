# Part 7 — Reference Appendix

## A.1 pdf-lib's drawText positioning system in more depth

`page.drawText`'s x/y coordinates specify the position of the text's baseline's left edge, in PDF's native point-based, bottom-left-origin coordinate system (the same origin convention discussed in Part 5's Section 5.4 for XFDF). This means increasing y moves text upward (toward the top of the page), the opposite of typical screen/browser coordinate conventions where increasing y moves downward — a detail easy to get backwards if you are used to CSS/Canvas coordinate systems, where Part 4's document-space Y axis deliberately matched screen convention specifically to avoid this exact confusion in that context. `rotate` accepts a `degrees()` or `radians()` value and rotates the text around its own origin point (the position you specified), which is why Part 7's watermark positions the text at `width/4, height/2` and then relies on the 45-degree rotation to sweep it across the page diagonally, rather than needing to manually calculate a rotated bounding box.

## A.2 AcroForm field types beyond text fields

Part 7's `forms.ts` only implements text field filling via `form.getTextField(name).setText(value)`, but pdf-lib's Form API supports every standard AcroForm field type: `getCheckBox(name).check()`/`.uncheck()` for checkboxes, `getRadioGroup(name).select(optionName)` for radio button groups, `getDropdown(name).select(optionName)` for dropdown/combo fields, and `getButton(name)` for push-button fields (typically used to trigger actions in interactive viewers, less relevant for our programmatic-filling use case). A more complete `fillFormFields` implementation would inspect each field's actual type (as `listFormFields` already reports via `field.constructor.name`) and dispatch to the correct type-specific setter rather than assuming every field is a text field — Part 7's implementation deliberately keeps to the single most common case (text fields) to keep the core teaching example focused, but extending it to handle checkboxes and dropdowns is a natural, well-supported next step using the exact same underlying `form.flatten()` mechanism for finalizing them.

## A.3 A deeper PDF/A conformance discussion

Building on Part 7's Section 7.4 scoping note: PDF/A actually has multiple conformance levels (PDF/A-1, PDF/A-2, PDF/A-3, each with sub-levels like "a," "b," and "u" indicating different strictness around things like Unicode text-extraction guarantees) — "PDF/A compliant" is not a single binary property but a family of related, increasingly strict standards. True conformance validation typically requires a dedicated validator tool (the open-source veraPDF project is a commonly used, ISO-endorsed validator) that checks dozens of specific structural rules beyond what any general-purpose PDF library like pdf-lib attempts to enforce. If Greymatter PDF's users genuinely need certified PDF/A output for regulatory purposes, the realistic path forward is: use Part 7's flattening as one necessary preparatory step, then either integrate a dedicated PDF/A conversion tool (some commercial and some open-source options exist as separate libraries/services) or use LibreOffice's own PDF/A export mode (`soffice --convert-to "pdf:writer_pdf_Export:{\"SelectPdfVersion\":{\"type\":\"long\",\"value\":\"1\"}}"` is one documented LibreOffice invocation for PDF/A-1 export specifically) as an additional conversion pass through the same Part 8 microservice pattern, rather than attempting to hand-roll full conformance validation inside pdf-lib.

## A.4 Glossary of terms introduced in Part 7

- **AcroForm**: a PDF's built-in system for interactive form fields (text boxes, checkboxes, dropdowns) layered on top of static page content.
- **Flattening**: permanently baking form field values into a page's static content and removing the interactive fields entirely.
- **Watermark**: semi-transparent text or graphics stamped across a document's pages, typically to indicate status (e.g. "DRAFT," "CONFIDENTIAL").
- **StandardFonts**: 14 fonts (Helvetica, Times Roman, Courier, and their bold/italic variants, plus Symbol and ZapfDingbats) guaranteed available in every PDF reader without embedding font data.
- **PDF/A**: an ISO-standardized subset of PDF specifically designed for long-term digital archival, restricting features that could cause future unreadability.
- **veraPDF**: an open-source, ISO-endorsed PDF/A conformance validator (mentioned as a real-world next step, not implemented in this series).

## A.6 Common pitfalls at this stage

1. **Assuming every PDF has form fields**: as Part 7's own verification steps note, most everyday PDFs have zero AcroForm fields — calling `flatten()` on such a document is harmless (it simply does nothing, since there is nothing to flatten) but testing form-filling specifically requires deliberately sourcing or creating a file that actually contains fillable fields.
2. **Confusing "filling" with "flattening"**: filling alone leaves a form still editable by the next person who opens the file — only explicitly calling `flatten()` afterward makes the values permanent; conflating these two steps is a common source of confusion when a "final" document unexpectedly remains editable.
3. **Getting the rotation origin point backwards**: since `rotate()` rotates around the (x, y) position you specify, moving that position without adjusting your mental model of where the rotated text will actually end up can produce watermarks that clip off the visible page unexpectedly — always test visually after any positioning change, rather than trusting the math alone.
4. **Forgetting that watermarking/filling/flattening all create new Document rows**: consistent with Part 6's pattern, expecting these operations to have modified the original document in place (rather than checking the document list for a newly-created entry) is a common point of confusion when first testing these features.
