# Part 2 — Reference Appendix

Deep-dive material for Part 2, isolated from the step-by-step tutorial flow.

## A.1 pdf.js internals: what actually happens inside getDocument() and render()

When pdfjs-dist's `getDocument({ data: arrayBuffer })` is called, it does not immediately parse the whole file. It creates a "loading task" that lazily reads the PDF's cross-reference table (a lookup index at the end of every PDF file mapping object numbers to byte offsets) and the document's trailer (metadata pointing at the root object). Only once you call `getPage(n)` does pdf.js parse that specific page's content stream — the sequence of drawing operators (move-to, line-to, fill, show-text, etc.) that describe what to paint. This lazy-loading design is why pdf.js can handle very large documents without parsing every page upfront: opening a 500-page PDF and viewing only page 1 does not require parsing pages 2–500 at all until they are actually requested.

`page.render({ canvasContext, viewport })` walks that page's content stream operator-by-operator, translating each PDF drawing primitive into equivalent Canvas 2D API calls (fillRect, fillText, drawImage, etc.) on the context you provide. This is genuinely CPU-intensive for pages with complex vector graphics or many small text runs, which is exactly why Part 2 moved this work into a Web Worker.

## A.2 Transferable objects: why ArrayBuffer and ImageBitmap avoid copying

By default, `postMessage()` between a main thread and a Worker clones data using the "structured clone algorithm" — a deep copy that can be slow for large objects, since every byte must be duplicated in memory. A **Transferable object** is a specific set of types (ArrayBuffer, ImageBitmap, MessagePort, and a few others) that postMessage can instead transfer by reference: ownership of the underlying memory moves from the sender to the receiver, with zero copying. After a transfer, the sending side's reference becomes unusable (its `byteLength` becomes 0 for an ArrayBuffer, for example) — this is a deliberate safety mechanism preventing both threads from simultaneously mutating the same memory.

This is why Part 2's `usePdfWorker.loadDocument` passes `[arrayBuffer]` as the transfer list when sending a `LOAD_DOCUMENT` message, and why the worker's `respond()` function passes `[bitmap]` when sending back a `PAGE_RENDERED` message — both the initial file upload buffer and each rendered page's bitmap can be tens of megabytes, and structured-clone-copying that volume of data on every single page render would introduce noticeable, unnecessary latency.

## A.3 OffscreenCanvas browser support notes

OffscreenCanvas, used inside `src/workers/pdf.worker.ts`, is supported in all current versions of Chrome, Edge, Firefox, and Safari (Safari added support later than the others, in Safari 16.4, released March 2023) — if you need to support meaningfully older browser versions than that, you would need a fallback path that renders on the main thread instead, sacrificing this Part's main-thread-blocking protection for those specific older browsers. As of the Next.js 16 / React 19.2 timeframe this series targets, OffscreenCanvas can be treated as a safe, broadly-available baseline for a new application.

## A.4 Why we pin pdfjs-dist's exact version

pdf.js's main package and its separate `pdf.worker.min.mjs` file are versioned together and communicate using an internal API version number embedded in both. If your main package version and worker script version drift apart (e.g. by upgrading pdfjs-dist without re-running the copy-pdf-worker script, or vice versa), pdf.js throws an explicit "API version mismatch" error rather than silently misbehaving — a deliberate, helpful failure mode, but one best avoided entirely by keeping the two in lockstep, which is why Part 2 pins an exact version (4.6.82) rather than a caret range (^4.6.82) that npm could otherwise resolve to a newer patch version on a fresh `npm install` without your worker script being correspondingly updated.

## A.5 Glossary of terms introduced in Part 2

- **Web Worker**: a background JavaScript execution thread provided by browsers, separate from the main thread handling UI events, with no direct DOM access.
- **OffscreenCanvas**: a canvas-like drawing surface usable inside a Web Worker (where no real DOM `<canvas>` element can exist).
- **Transferable object**: a type (ArrayBuffer, ImageBitmap, etc.) that postMessage can move by reference instead of deep-copying, for performance.
- **devicePixelRatio**: the ratio between physical screen pixels and CSS pixels, commonly 1, 2, or 3 on modern displays; used to render bitmaps sharply on high-density screens.
- **Content stream**: the sequence of drawing operators inside a PDF page describing what to paint (text, lines, images, fills).
- **Cross-reference table**: an index at the end of a PDF file mapping internal object numbers to byte offsets, allowing readers to jump directly to any object without scanning the whole file.

## A.7 Common pitfalls at this stage

1. **Forgetting to re-run copy:pdf-worker after upgrading pdfjs-dist**: causes the API version mismatch error described in A.4.
2. **Instantiating the Worker without `{ type: "module" }`**: since `src/workers/pdf.worker.ts` uses ES module `import` syntax, omitting this option causes a syntax error when the browser tries to load it as a classic (non-module) script.
3. **Forgetting to call `bitmap.close()`**: ImageBitmap objects hold GPU/CPU memory that should be explicitly released once painted; forgetting this across many page renders can gradually increase memory usage in long viewing sessions.
4. **Testing devicePixelRatio-dependent behavior in a non-Retina environment**: on a standard 1x display, devicePixelRatio is 1, so bugs in the ×2 or ×3 scaling logic may not surface until tested on a genuinely high-density screen.
