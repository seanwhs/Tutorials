# Part 6 — Reference Appendix

Deep-dive material for Part 6, isolated from the step-by-step tutorial flow.

## A.1 pdf-lib's API surface in more depth

Beyond the `PDFDocument.create`/`load`/`save`, `copyPages`, and `addPage`/`getPageCount`/`getPageIndices` methods used directly in Part 6, pdf-lib exposes a substantial additional API used in later parts and worth knowing exists: `embedFont` (Part 7, for watermark text), `embedPng`/`embedJpg` (for embedding raster images, e.g. a company logo stamp), `drawText`/`drawImage`/`drawRectangle`/`drawLine` (low-level content-stream drawing primitives, the same category of API used for watermarking in Part 7), `getForm` (Part 7, AcroForm access), and `setTitle`/`setAuthor`/`setSubject` (document metadata). pdf-lib deliberately does not expose a rendering/rasterization API (turning a page into pixels) — that remains pdf.js's job (Part 2); pdf-lib is exclusively a structural read/write library.

## A.2 Memory considerations for very large PDF merges

Section 9.2 and 9.4 (Part 9) discuss production-hardening safeguards for this concern in depth, but it is worth previewing the underlying mechanism here: `PDFDocument.load()` parses a PDF's entire object graph (every page, font, image, and structural element) into an in-memory JavaScript object representation — this representation is typically larger than the original file's byte size, sometimes substantially so for files with many small objects, because each PDF object becomes a full JavaScript object with associated overhead, rather than staying as compact serialized bytes. `copyPages` during a merge operation performs a deep copy of every referenced resource (fonts, images) from the source into the destination document's own object graph, meaning a merge of N documents holds, at its peak, roughly the combined in-memory representation of all N source documents plus the growing destination document simultaneously — this is the concrete mechanism behind Part 9's decision to cap the number of documents mergeable in a single operation.

## A.3 Why copyPages deep-copies fonts and images rather than referencing them

You might wonder why merging two PDFs that happen to use the same font (e.g. both use standard Helvetica) results in that font's data being embedded twice in the merged output, rather than pdf-lib being clever enough to deduplicate it. The reason is that PDF's internal object model uses per-document object numbering — an object numbered "5" in document A has no relationship to object "5" in document B; they are entirely separate numbering namespaces. When merging, pdf-lib must copy every object a page depends on into the destination document's own namespace, assigning it fresh object numbers there, because simply pointing at "object 5" in the output would be ambiguous once multiple source documents are involved. Some PDF tools implement more sophisticated deduplication (detecting that two embedded font resources are byte-identical and only storing one copy), but pdf-lib's simpler always-copy approach favors correctness and implementation simplicity over minimizing output file size — a deliberate, reasonable tradeoff for a general-purpose library.

## A.4 Glossary of terms introduced in Part 6

- **pdf-lib**: a pure JavaScript library for creating and structurally modifying PDF files (the "writer," complementing pdf.js the "reader").
- **Object graph**: the network of interconnected internal PDF objects (pages, fonts, images, content streams) that together define a PDF's structure.
- **copyPages**: pdf-lib's method for deep-copying pages (and everything they depend on) from one PDFDocument into another.
- **FormData**: a web-standard object representing form/file data for an HTTP request body, used both for Next.js Server Action file uploads and traditional HTML form submissions.
- **Server Action**: *(recap from Part 1)* an async function marked `"use server"`, callable directly from a Client Component as though local, executing securely on the server.

## A.6 Common pitfalls at this stage

1. **Forgetting `await` on `.save()`**: pdf-lib's `save()` is asynchronous (it returns a Promise) even though the operation feels conceptually synchronous — forgetting to await it produces a Promise object where byte data was expected, which typically surfaces as a confusing storage-layer error rather than an obviously-related one.
2. **Passing 1-indexed page numbers directly to pdf-lib APIs**: pdf-lib's own page indices are 0-indexed; Greymatter PDF's own convention (matching pdf.js) is 1-indexed — the explicit `pageNumber - 1` conversion in `extractPages` is not optional boilerplate, it is a necessary translation between two different, legitimate conventions.
3. **Assuming merge preserves annotations**: Part 6's `mergeDocuments` copies page *content* but does not carry over Part 5's database-stored Annotation records, since those are associated with the original `documentId`, not with the pages themselves — a reader wanting merged documents to also carry forward their source annotations would need to explicitly write additional logic re-associating (and re-indexing, since page numbers shift during a merge) annotation records with the new merged document.
4. **Not handling the "document not found" case distinctly from "access denied"**: consistent with Part 3's Appendix A.6 point 2, `loadDocumentBytes`'s error messages should avoid leaking which `documentId` values exist to an unauthorized caller.
