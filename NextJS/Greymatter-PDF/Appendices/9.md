# Part 9 — Reference Appendix

Deep-dive material for Part 9, isolated from the step-by-step tutorial flow. This is also the final appendix in the Greymatter PDF series.

## A.1 Next.js caching strategies beyond `use cache`

Part 9 applied function-level `use cache` to `listFormFields`, but Next.js's broader caching system includes several other layers worth knowing about for future work: the **Data Cache** (persistent caching of `fetch()` calls across requests and deployments, configurable via fetch's own cache/revalidate options), the **Full Route Cache** (caching entire rendered route output for routes with no dynamic data dependencies), and the **Router Cache** (a client-side, in-memory cache of recently visited route segments, reducing redundant server round-trips during client-side navigation). `use cache` (the directive Part 9 used) is Next.js 16's newer, more granular mechanism specifically for caching individual function outputs (like our `listFormFields`) rather than whole routes, and `unstable_cacheLife`/`unstable_cacheTag` (the latter not used in this series, but relevant for future work) let you fine-tune exactly how long cached data persists and how to explicitly invalidate it when underlying data does change unexpectedly.

## A.2 Cache invalidation: the hard problem you should know about even though we avoided it here

Part 9's caching choice was deliberately conservative specifically to sidestep cache invalidation's genuine difficulty: because `listFormFields`'s underlying data (a specific `documentId`'s bytes) is provably immutable for that ID's entire lifetime, we never need to manually invalidate its cache entry — it simply expires naturally after the "days" lifetime hint, harmlessly, since the data could never have changed anyway. Contrast this with a hypothetical cached "list of a user's documents" function: that data changes every time a new document is uploaded, so caching it naively would show stale results until the cache's lifetime expires, actively confusing users who just uploaded something and do not see it appear. Correctly caching genuinely mutable data requires either short cache lifetimes (accepting some staleness as a tradeoff for reduced load) or explicit invalidation (calling `revalidateTag`/`revalidatePath`, or Next.js 16's cache-tag mechanisms, at the exact moment the underlying data changes, e.g. inside `uploadDocument` right after a successful upload) — a meaningfully harder problem than Part 9's chosen example, included here so you understand why Part 9 specifically picked an example that avoids needing to solve it.

## A.3 Error Boundary patterns beyond route-level error.tsx

Next.js's route-level `error.tsx` (used in Part 9) is the coarsest-grained Error Boundary option — it catches anything unhandled anywhere within that route segment. React also supports manually-authored, more finely-scoped Error Boundaries via a class component implementing `static getDerivedStateFromError` and `componentDidCatch` (Error Boundaries currently cannot be written as function components using hooks alone; this remains one of the few cases in modern React where a class component is still the correct, necessary tool). A production-grade Greymatter PDF might wrap just the `PdfPageCanvas` component (rather than the whole route) in its own dedicated Error Boundary, so a rendering failure on one specific page of a multi-page document shows an inline "this page failed to render" message for just that one page, while every other page's canvas continues rendering normally alongside it — an even finer-grained isolation boundary than Part 9 implemented, and a natural extension exercise.

## A.4 Profiling and load-testing considerations beyond process.memoryUsage()

Part 9's `process.memoryUsage()` logging is a simple, immediately actionable technique for spot-checking a specific operation's memory footprint during development, but a genuine production readiness assessment would go further: Node.js's built-in `--inspect` flag (or tools like Clinic.js) enable full CPU/memory profiling sessions capturing exactly which functions consume the most time/memory across a realistic workload, and dedicated load-testing tools (k6, Artillery, or similar) let you simulate many concurrent users performing realistic sequences of actions (uploading, viewing, annotating, merging) to observe how the whole system — not just one isolated function — behaves under sustained concurrent load, which is the load pattern that actually matters for a real deployment, as opposed to any single operation's cost in isolation.

## A.5 Glossary of terms introduced in Part 9

- **`use cache`**: a Next.js 16 directive marking a function's return value for caching, keyed automatically by its arguments.
- **Cache invalidation**: the problem of ensuring cached data is updated or discarded once the underlying source data changes, to avoid serving stale results.
- **Error Boundary**: a React mechanism (route-level via `error.tsx`, or manually via a class component) that catches rendering-time errors in its children and displays a fallback UI instead of unmounting the whole tree.
- **RSS (Resident Set Size)**: the total physical memory currently allocated to a process by the operating system, as reported by `process.memoryUsage()`.
- **Heap**: the memory region where a JavaScript engine (V8, in Node.js's case) allocates objects; `heapUsed` specifically reports how much of that region is currently occupied by live objects.

## A.7 Common pitfalls at this stage

1. **Caching data that is not actually immutable**: always explicitly verify (as Part 9's Section 9.1 does in writing) that a function's output truly cannot change for a given set of arguments before applying `use cache` — caching genuinely mutable data without an invalidation strategy produces confusing, hard-to-diagnose staleness bugs.
2. **Relying solely on the framework-level body size limit**: Part 1's Server Action `bodySizeLimit` config is a useful first layer, but Part 9's explicit `assertFileSizeAllowed` check provides a clearer, more specific error message and ensures consistent enforcement across every upload path, rather than depending entirely on one global setting.
3. **Forgetting that Error Boundaries do not catch async callback errors**: as Section 9.8 explicitly discusses, an error thrown inside a Promise rejection or a Web Worker's `onerror` handler will not be caught by a React Error Boundary unless your own code explicitly catches it first (as Part 2's `PdfPageCanvas` already correctly does) — assuming `error.tsx` alone provides complete crash protection is a dangerous, incorrect assumption.
4. **Treating memory usage numbers from a single local test as representative of production behavior**: a single merge operation's memory profile on your development machine, with no concurrent load, does not necessarily predict how the same code behaves under many simultaneous production requests — treat Section 9.5's technique as a useful development-time sanity check, not a substitute for genuine load testing before a real production launch.
