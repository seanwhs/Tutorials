## A.1 SVG vs Canvas tradeoffs in more depth

Part 4 chose SVG for the annotation overlay, and it is worth being precise about why, and when the opposite choice would make more sense. SVG represents each shape as a distinct DOM node (an actual `<rect>`, `<path>`, etc. element the browser tracks individually) — this gives you free hit-testing (clicking directly on a specific shape naturally fires events on that shape's element), free accessibility hooks (you can attach `aria-label` to individual shapes), and trivial selective updates (changing one annotation's color only touches that one DOM node). The cost is that SVG performance degrades as the number of individual elements grows very large — documents with many thousands of annotations on a single page could start to feel sluggish, since the browser must manage that many individual DOM nodes.

Canvas, by contrast, is a single element regardless of how much you draw inside it — excellent raw rendering performance even with huge numbers of shapes, but you lose per-shape hit-testing and accessibility for free; you would need to manually implement your own hit-testing logic (checking mouse coordinates against your own list of shape bounding boxes) to know which shape, if any, was clicked. For Greymatter PDF's expected use case (dozens, perhaps low hundreds of annotations per page, not tens of thousands), SVG's developer-ergonomics advantages outweigh Canvas's raw throughput advantage. If a future version of this product needed to support extremely annotation-dense documents, revisiting this tradeoff toward a Canvas-based overlay (with manually implemented hit-testing) would be a reasonable, informed choice at that point.

## A.2 Pointer Events vs Mouse Events browser compatibility

`AnnotationOverlay` uses `onPointerDown`/`onPointerMove`/`onPointerUp` rather than the older `onMouseDown`/`onMouseMove`/`onMouseUp`. The Pointer Events API unifies mouse, touch, and stylus/pen input behind one consistent event model — using it means Greymatter PDF's drawing interactions work correctly on a touchscreen tablet with a stylus, or a touch-only device, without any separate touch-specific event handling code. Pointer Events have been supported in all major browsers for several years as of this series' target timeframe, so no polyfill or fallback is necessary for a new application. The one behavioral detail worth knowing: `setPointerCapture` (used in `handlePointerDown`) ensures the element that started the interaction continues receiving move/up events even if the pointer moves outside that element's bounds mid-drag — without this, a fast drag that briefly exits the SVG's rectangle could "lose" the drag partway through.

## A.3 A deeper look at the React Compiler's memoization in AnnotationOverlay

To make the Part 1 promise concrete: without the React Compiler, a hand-optimized version of `AnnotationOverlay` would likely wrap the `annotations.filter(...).map(...)` block in a `useMemo` with dependencies `[annotations, pageNumber, pageWidthCss, pageHeightCss]`, and wrap `handlePointerDown`/`handlePointerMove`/`handlePointerUp` each in their own `useCallback`. The React Compiler's static analysis examines the component's code at build time and inserts semantically equivalent memoization automatically — it can determine, by analyzing which values a given expression actually depends on, exactly when it is safe to skip recomputing that expression on a given render, without a developer needing to enumerate dependency arrays by hand (a notoriously easy place to introduce subtle bugs, e.g. by forgetting a dependency and causing stale closures).

It is worth being honest about a limitation: the Compiler cannot help with genuinely necessary state updates — every actual pointer-move event during a drag still triggers a real React re-render (because `draftRect`'s state genuinely changed), and that specific re-render's cost cannot be memoized away, because the whole point is that the visible preview rectangle must visually update as the user drags. What the Compiler removes is only the *unnecessary* re-computation of things that did not need to change on a given render, not the necessary work of reflecting real state changes on screen.

## A.4 Glossary of terms introduced in Part 4

- **Document space**: coordinates expressed as fractions (0 to 1) of a PDF page's total width/height, independent of zoom level or screen size.
- **Screen space**: coordinates expressed in actual on-screen CSS pixels, which change whenever zoom level or window size changes.
- **SVG (Scalable Vector Graphics)**: an XML-based, DOM-integrated vector graphics format where each shape is an individually addressable element.
- **Pointer Events**: a unified browser event API covering mouse, touch, and stylus input with one consistent model.
- **setPointerCapture**: a DOM method ensuring an element continues receiving pointer events even after the pointer moves outside its bounds, until the interaction ends.
- **React Compiler**: a build-time tool (introduced in Part 1) that automatically inserts memoization equivalent to manual `useMemo`/`useCallback`, based on static analysis of component code.

## A.6 Common pitfalls at this stage

1. **Forgetting to convert coordinates at the right time**: always convert screen-to-document space at the moment an annotation is *created* (inside `handlePointerUp`), and document-to-screen space at the moment it is *rendered* (inside the JSX map) — mixing these up produces annotations that appear correctly once but drift on the next zoom change.
2. **Testing only at one zoom level**: a coordinate bug that appears identical at 100% zoom can be invisible until you explicitly test at a different zoom level (as Part 4's Step 6 verification deliberately does) — always include a zoom-change test when touching annotation positioning code.
3. **Missing `pointerEvents: "none"` when drawing is disabled**: without this, the invisible SVG overlay still intercepts clicks even when the user is not actively drawing, silently breaking any future interaction (e.g. text selection) that should otherwise reach the page beneath it.
4. **Forgetting the `key` prop on mapped SVG elements**: as with any React list rendering, omitting `key={annotation.id}` can cause subtle rendering bugs when annotations are added/removed, since React cannot correctly track which DOM node corresponds to which annotation.
