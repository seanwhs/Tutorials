# Next.js 16 TypeScript Foundations

## Part 1 — RootLayout Made Simple: Understanding the Most Confusing File in Your Next.js App

> **Series:** From Confusing Layout Syntax to Confident Typed App Router Code

You’ve just run `npx create-next-app@latest` with TypeScript and opened `app/layout.tsx`. At first glance, it can feel like a lot is happening at once: a default export, typed props, `children`, `React.ReactNode`, raw `<html>` and `<body>` tags, and an optional `metadata` export. The good news is that almost none of this is magic once you see how the Root Layout works in the App Router. [medium](https://medium.com/@rbhanawat328/next-js-app-router-a-complete-guide-410dc5e1faf9)

```tsx
import type { Metadata } from 'next'
import { Inter } from 'next/font/google'
import './globals.css'

const inter = Inter({ subsets: ['latin'] })

export const metadata: Metadata = {
  title: 'My Next.js App',
  description: 'Generated by create next app',
}

export default function RootLayout({
  children,
}: Readonly<{
  children: React.ReactNode
}>) {
  return (
    <html lang="en">
      <body className={inter.className}>{children}</body>
    </html>
  )
}
```

### What `layout.tsx` does

In the App Router, `layout.tsx` defines UI that wraps one or more pages and stays in place as you navigate. That makes it ideal for shared structure such as headers, navigation, footers, fonts, global CSS, and providers. In practical terms, the layout is the frame and the current route’s page is the content that changes inside it. [nextjs](https://nextjs.org/learn/dashboard-app/adding-metadata)

A typical route structure looks like this:

```text
app/
├── layout.tsx
├── page.tsx
├── about/
│   └── page.tsx
└── products/
    └── page.tsx
```

When the user visits `/about` or `/products`, the root layout remains the outer shell while the matching page is injected into it through `children`. Next.js also preserves layouts during client-side navigation so shared UI does not reset unnecessarily. [medium](https://medium.com/@rbhanawat328/next-js-app-router-a-complete-guide-410dc5e1faf9)

### Why the root layout is required

The file `app/layout.tsx` is the root layout, and Next.js requires every app to have one. This is the top-most layout in the app directory, and it is where you define the document-level `<html>` and `<body>` tags. Unlike nested layouts, the root layout is responsible for the overall HTML document shape, not just a section of the UI. [nextjs](https://nextjs.org/learn/dashboard-app/adding-metadata)

That means this file is the right place for global concerns such as:
- global CSS.
- shared fonts.
- top-level metadata.
- site-wide providers.
- accessibility attributes like `lang`. [medium](https://medium.com/@rbhanawat328/next-js-app-router-a-complete-guide-410dc5e1faf9)

### The core idea: `children`

`children` is the most important prop in the file, and it is not special to Next.js — it is standard React. Any component can accept `children`, and React automatically passes the content nested inside it through that prop. [nextjs](https://nextjs.org/learn/dashboard-app/adding-metadata)

```tsx
function Card({ children }) {
  return <div className="card">{children}</div>
}
```

Usage:

```tsx
<Card>
  <h1>Hello World</h1>
  <p>This content becomes children.</p>
</Card>
```

In Next.js, the framework handles that same pattern for you. Conceptually, the current page gets passed into the layout as `children`, and the layout decides where that content should appear. That is why your page content usually belongs inside `<main>{children}</main>`. [medium](https://medium.com/@rbhanawat328/next-js-app-router-a-complete-guide-410dc5e1faf9)

### Why `<html>` and `<body>` appear here

In most React projects, the HTML document lives outside your components. In the App Router, the root layout is the document wrapper, so it must render `<html>` and `<body>` itself. You should not repeat those tags in nested layouts because only the root layout owns the document shell. [medium](https://medium.com/@rbhanawat328/next-js-app-router-a-complete-guide-410dc5e1faf9)

This is also why the Metadata API is preferred over manually placing `<title>` and `<meta>` tags in the root layout. Next.js handles head management, deduplication, and streaming-friendly behavior for you. [medium](https://medium.com/@rbhanawat328/next-js-app-router-a-complete-guide-410dc5e1faf9)

### What `React.ReactNode` means

`React.ReactNode` means “anything React can render,” including elements, text, fragments, arrays, and `null`. It is a natural fit for `children` because `children` can contain many different renderable values. In other words, it is flexible without being vague. [medium](https://medium.com/@rbhanawat328/next-js-app-router-a-complete-guide-410dc5e1faf9)

You may also see the same type written as `ReactNode` imported from `react`. Both are common, and the important part is that the prop is typed as renderable React content.

### Reading the full signature

```tsx
export default function RootLayout({          // Default export component
  children,                                   // Destructured prop
}: Readonly<{                                 // Readonly props wrapper
  children: React.ReactNode                   // Renderable child content
}>) {
  return (
    <html lang="en">
      <body>{children}</body>                // Current route content
    </html>
  )
}
```

That entire file is just a React component with a special filename and a special job. Once you see that, the confusion drops away quickly. [nextjs](https://nextjs.org/learn/dashboard-app/adding-metadata)

## A better mental model

Think of the root layout as the app’s permanent shell. The shell stays, and the page content swaps inside it when the route changes. That is why layouts are so powerful: they let you avoid repeating shared UI and keep navigation smooth. [medium](https://medium.com/@rbhanawat328/next-js-app-router-a-complete-guide-410dc5e1faf9)

A simple way to remember it is this:

> Layout = persistent frame.  
> Page = route-specific content.  
> `children` = the content slot between them.

## Common mistakes

- Forgetting `{children}`, which causes your page content not to appear.
- Adding `<html>` and `<body>` inside nested layouts, which should only happen in the root layout. [medium](https://medium.com/@rbhanawat328/next-js-app-router-a-complete-guide-410dc5e1faf9)
- Treating `children` as a Next.js-only idea, when it is really a standard React pattern. [nextjs](https://nextjs.org/learn/dashboard-app/adding-metadata)
- Hardcoding page components inside the layout instead of letting Next.js inject them through `children`.
- Putting heavy client-side logic directly in the root layout when a deeper client component or provider would be a better fit.

## A practical version

```tsx
import type { Metadata } from 'next'
import { Inter } from 'next/font/google'
import './globals.css'

const inter = Inter({ subsets: ['latin'] })

export const metadata: Metadata = {
  title: 'My App',
  description: 'Built with Next.js 16',
}

export default function RootLayout({
  children,
}: Readonly<{
  children: React.ReactNode
}>) {
  return (
    <html lang="en">
      <body className={inter.className}>
        <header>
          <nav>Navigation</nav>
        </header>
        <main>{children}</main>
        <footer>© 2026 My Company</footer>
      </body>
    </html>
  )
}
```

This keeps your global structure in one place while every page still renders normally inside the layout. [medium](https://medium.com/@rbhanawat328/next-js-app-router-a-complete-guide-410dc5e1faf9)
