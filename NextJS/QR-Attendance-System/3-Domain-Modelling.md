# Production Patterns with Next.js 16

## Reference Architecture #1 — Part 3

# Modeling Events and Attendance with Sanity

### Designing a Flexible Domain Model for Durable Workflows

> **The quality of a workflow depends on the quality of the data model behind it.**

---

# Executive Summary

In Part 2, we established the project's structure and architectural boundaries.

Now it's time to design the domain itself.

A common mistake when building attendance applications is to model attendance as a simple boolean:

```
checkedIn = true
```

While sufficient for small applications, this approach quickly becomes limiting.

Production systems need to answer questions such as:

* When did the attendee check in?
* Which device was used?
* Was the request retried?
* Was the attendee checked in offline?
* Which entrance was used?
* Who manually approved the attendee?
* Which workflow execution created this record?

These questions require a richer domain model.

In this article we'll design the Sanity schemas that support both today's requirements and tomorrow's features.

---

# Domain Model

Instead of thinking in collections or tables, begin with business concepts.

```text id="domain-overview"
                     Event
                       │
         ┌─────────────┴─────────────┐
         ▼                           ▼
 Check-in Window              QR Configuration
         │
         ▼
 Attendance Record
         │
         ▼
 Clerk User
```

Each document represents a real business concept.

---

# Why Sanity?

Many developers associate Sanity primarily with marketing websites.

However, Sanity's document model also makes it well suited to operational applications where schemas evolve over time.

For an attendance platform, that flexibility provides several advantages:

* Structured content with strong validation
* Reference-based relationships
* Rich querying with GROQ
* Draft and published content
* Extensible document schemas
* First-class support for metadata

The event definition becomes content managed by organizers, while attendance records become operational documents generated by workflows.

---

# Event Schema

The event document describes everything required to run an event.

Typical fields include:

| Field           | Purpose                  |
| --------------- | ------------------------ |
| Title           | Display name             |
| Slug            | Public URL               |
| Description     | Event details            |
| Venue           | Physical location        |
| Starts At       | Event schedule           |
| Ends At         | Event schedule           |
| Check-in Opens  | Attendance window        |
| Check-in Closes | Attendance window        |
| Capacity        | Operational planning     |
| Status          | Draft, Published, Closed |

Notice that the check-in window is independent of the event schedule.

This allows organizers to open check-in before the event begins and close it after the scheduled start time.

---

# QR Configuration

Instead of hardcoding QR behaviour, model it explicitly.

Suggested fields include:

| Field              | Purpose                   |
| ------------------ | ------------------------- |
| Signed Token       | Prevent URL tampering     |
| Expiration         | Time-bound access         |
| Geofencing Enabled | Optional venue validation |
| Radius             | Allowed distance          |
| One-Time Use       | Optional security         |
| Manual Override    | Operational flexibility   |

Embedding these settings in the event document allows different events to enforce different policies without requiring application changes.

---

# Attendance Record

The attendance record is the most important document in the system.

Unlike a traditional join table, it captures the complete context of the check-in.

Recommended fields include:

| Field           | Purpose                      |
| --------------- | ---------------------------- |
| Event Reference | Associated event             |
| Clerk User ID   | Authenticated attendee       |
| Checked In At   | Server timestamp             |
| Status          | Attendance lifecycle         |
| Method          | QR, Manual, Offline          |
| Device          | Browser or mobile device     |
| IP Address      | Audit trail (if appropriate) |
| Workflow ID     | Traceability                 |
| Retry Count     | Operational diagnostics      |
| Correlation ID  | Cross-system tracing         |

The richer the record, the easier it becomes to support reporting, troubleshooting, and compliance.

---

# Attendance Lifecycle

Attendance should be modelled as a state machine rather than a boolean.

```text id="attendance-state-machine"
Pending
   │
   ▼
Validated
   │
   ▼
Checked In
   │
   ├────────────►Confirmed
   │
   └────────────►Cancelled
```

This approach leaves room for future features such as approvals, badge printing, or post-event reconciliation without requiring disruptive schema changes.

---

# Idempotency

One attendee should only ever have one attendance record per event.

Rather than relying solely on database constraints, derive a deterministic idempotency key from the authenticated user and the event.

```text id="idempotency-key"
SHA256(eventId + ":" + userId)
```

Every workflow checks for an existing record using this key before creating a new document.

Repeated requests converge on the same result instead of creating duplicates.

---

# Audit Metadata

Operational metadata is often overlooked until something goes wrong.

Include fields such as:

* Workflow execution ID
* Correlation ID
* Retry attempts
* Processing duration
* Source (Web, Mobile, Offline Sync)
* User agent

These values become invaluable when investigating production issues.

---

# Query Patterns

Before finalizing a schema, consider the queries you'll need.

Examples include:

* Has this attendee already checked in?
* How many attendees are currently checked in?
* Which attendees arrived after the keynote?
* Which check-ins were completed offline?
* Which workflows required retries?
* Which entrances experienced the highest traffic?

Designing for these questions now avoids painful migrations later.

---

# Engineering Decision Record — Attendance as a Document

**Problem**

Should attendance be stored inside the event document?

**Decision**

Store attendance as independent documents.

**Benefits**

* Scales independently
* Supports efficient querying
* Simplifies auditing
* Enables event replay
* Keeps event metadata immutable

**Trade-offs**

Requires references between documents, but dramatically improves scalability and flexibility.

---

# Common Modeling Mistakes

Avoid these shortcuts:

* Storing `checkedIn: true` on the user.
* Embedding attendance inside the event document.
* Trusting timestamps generated by the client.
* Omitting workflow metadata.
* Ignoring idempotency.
* Coupling persistence models to UI requirements.

These decisions often seem harmless early on but become costly as the system evolves.
